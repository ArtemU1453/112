# Engineering Handbook — ЕАСУР

- **Version:** 1.0
- **Status:** APPROVED

Единые инженерные процессы проекта. Обязателен для всех участников, включая ИИ.

## Section 1. Общие принципы

Цель — промышленная государственная ИС. Принципы: качество выше скорости; архитектура выше
реализации; безопасность по умолчанию; документация — часть продукта; каждое изменение
воспроизводимо; одна задача — один завершённый результат.

## Section 2. Жизненный цикл задачи

```
Backlog → Ready → In Progress → Code Complete → Code Review → Testing → Documentation Updated → Done
```

Пропуск стадий запрещён.

## Section 3. Работа с Backlog

Каждая задача: уникальный идентификатор, название, описание, бизнес-ценность, техническое
описание, DoR, DoD, зависимости, приоритет, оценка сложности, ссылки на Epic/Feature
(см. `docs/standards/backlog-structure.md`).

## Section 4. Ветвление

Одна задача — одна ветка. Именование: `feature/INC-001-short-description`, `bugfix/INC-125`,
`hotfix/AUTH-001`, `release/v1.0.0` (см. Git Strategy). Несколько независимых задач в одной
ветке запрещены.

## Section 5. Правила коммитов

Атомарные коммиты, Conventional Commits: `<type>(<scope>): <description>`. Типы: feat, fix,
refactor, perf, docs, test, build, ci, style, chore, revert. Примеры:
`feat(incident): add incident aggregate`, `fix(auth): validate refresh token`.

## Section 6. Code Review

Перед завершением — обязательный внутренний ревью: архитектура, читаемость, соответствие
стандартам, обработка ошибок, безопасность, производительность, тестируемость, отсутствие
дублирования, соответствие ADR/RFC.

## Section 7. Тестирование

Уровни (по применимости): Unit, Integration, Contract, End-to-End, Performance. Отсутствие
тестов должно быть явно обосновано требованиями (см. `docs/testing/`).

## Section 8. Документация

После изменения — определить необходимость обновления: Vision, Architecture, ADR, RFC, API,
Database, Security, Deployment, User Guide, Developer Guide. Требуемое обновление входит в
текущую задачу.

## Section 9. Работа ИИ (перед задачей)

Проверить: Project Constitution, Architecture Baseline, ADR, RFC, Coding Standards, открытые
зависимости, Definition of Ready. Только затем приступать к реализации.

## Section 10. Проверка перед завершением

Подтвердить: требования выполнены; архитектура не нарушена; документация актуальна; тесты
обновлены; зависимости проверены; безопасность проверена; известные ограничения описаны.

## Section 11. Работа с рисками

Выявленные архитектурные риски, риски безопасности/производительности и техдолг фиксируются
отдельной записью Backlog и связываются с текущей задачей.

## Section 12. Журнал решений

После задачи фиксируется: выполненная задача, изменённые компоненты, затронутые ADR/RFC,
изменения API и модели данных, изменения документации, выявленные риски, принятые технические
решения.

## Section 13. Непрерывное улучшение

После каждого этапа — Engineering Review: соблюдение стандартов, качество архитектуры, техдолг,
актуальность документации, эффективность процесса. Изменения Handbook — только через RFC с
сохранением совместимости с Constitution и Baseline.

## Section 14. Правила для ИИ

ИИ обязан: выполнять только текущую Task; не изменять несвязанные модули; не менять архитектуру
без утверждения; не придумывать отсутствующие требования; явно отмечать допущения; останавливать
выполнение при противоречии требований и архитектуры; считать Handbook обязательным.

## Поток создания ценности

```
Требование → Feature → User Story → Task → Код → Unit Test → Integration Test → Документация → Релиз
```
