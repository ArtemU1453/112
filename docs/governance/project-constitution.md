# Project Constitution — ЕАСУР

- **Version:** 1.0
- **Status:** APPROVED

Данный документ определяет фундаментальные правила разработки проекта. Правила обязательны
для всех участников, включая искусственный интеллект. Ни один документ проекта не может
противоречить Конституции.

## Single Source of Truth (порядок приоритета)

Архитектурная документация имеет приоритет над кодом. При противоречии руководствоваться
порядком:

1. Vision
2. Architecture Principles
3. Architecture.md
4. ADR
5. RFC
6. Domain Model
7. API Contracts (OpenAPI)
8. Database Schema
9. Coding Standards
10. Исходный код
11. Комментарии в коде

Если изменение нарушает хотя бы один утверждённый архитектурный документ, оно не может
считаться завершённым.

## Статьи Конституции

**ARTICLE 1 — Архитектура превыше кода.** Код соответствует архитектуре. Архитектура не
меняется через код: любое изменение начинается с RFC и завершается ADR.

**ARTICLE 2 — Единый источник истины.** Источник истины — утверждённые архитектурные
документы; код — их реализация, не замена.

**ARTICLE 3 — Production First.** Каждый модуль — production-ready. Запрещены: демонстрационный
код, временные решения, неполные реализации, заглушки, TODO, фиктивные зависимости,
неиспользуемый код.

**ARTICLE 4 — Инкрементальная разработка.** Проект — из небольших завершённых задач. Каждая
Task имеет DoR, DoD, архитектурное соответствие, тесты, документацию.

**ARTICLE 5 — Качество превыше скорости.** Нельзя сокращать проверки ради скорости. Любой код
проходит Code Review, Static Analysis, Security Review, Testing, Documentation Review.

**ARTICLE 6 — Обратная совместимость.** Изменения не ломают функциональность без утверждённого
RFC. При неизбежном нарушении — последствия, план миграции, план отката.

**ARTICLE 7 — Единые стандарты.** Единые именование, логирование, обработка ошибок, структура
каталогов, архитектурные слои, стиль кода, версионирование API.

**ARTICLE 8 — Тестирование обязательно.** Функциональность без тестов считается незавершённой.

**ARTICLE 9 — Документация — часть проекта.** Любое изменение сопровождается обновлением
документации; документация не отстаёт от реализации.

**ARTICLE 10 — Безопасность по умолчанию.** Security by Design: минимальные привилегии,
шифрование, аудит, безопасная обработка ошибок, безопасное хранение секретов.

**ARTICLE 11 — Наблюдаемость по умолчанию.** Каждый модуль: структурированное логирование,
метрики, трассировка, мониторинг, health checks.

**ARTICLE 12 — API First.** Внешние интерфейсы сначала описываются контрактами; реализация —
после утверждения API.

**ARTICLE 13 — Database First.** Изменения модели данных сначала описываются документацией и
миграциями, затем изменяется код.

**ARTICLE 14 — Контролируемые изменения.** Изменение архитектуры — только после RFC, Impact
Analysis, ADR, Migration Plan, Rollback Plan.

**ARTICLE 15 — Единый Backlog.** Вся работа — через утверждённый Backlog (Program, Epic, Feature,
Story, Task, Bug, Spike, Technical Debt).

**ARTICLE 16 — Одна задача — один инкремент.** За один запуск — одна Task; самостоятельный
переход к следующей задаче запрещён.

**ARTICLE 17 — Обязательный Code Review.** Перед завершением задачи — внутренний ревью:
архитектура, качество, безопасность, производительность, соответствие стандартам.

**ARTICLE 18 — Непрерывная синхронизация.** После задачи проверяется актуальность Vision,
Architecture.md, ADR, RFC, OpenAPI, Database/Deployment/Security/Testing Documentation;
при необходимости обновляется в рамках той же задачи.

**ARTICLE 19 — Контроль технического долга.** Любой техдолг регистрируется отдельной задачей
Backlog; не скрывается и не игнорируется.

**ARTICLE 20 — ИИ подчиняется Конституции.** Решения ИИ соответствуют Конституции. При
конфликте запроса пользователя с Конституцией ИИ обязан: описать противоречие, объяснить
последствия, предложить безопасный вариант, не менять фундаментальные правила без отдельного
утверждения.

## Правило для ИИ

Если задача требует изменения утверждённой архитектуры, ИИ обязан остановить реализацию,
описать необходимость изменения, подготовить RFC и ADR, описать влияние на систему и дождаться
утверждения архитектурного решения. Архитектурная целостность приоритетнее скорости разработки.
