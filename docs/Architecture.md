# Architecture.md — ЕАСУР

> Приоритет SSOT: 3. Ниже Vision и Architecture Principles, выше ADR по общей структуре,
> но конкретные технологические решения фиксируются в ADR (см. `docs/adr/`).
>
> Статус: APPROVED · Baseline: v1.0

Документ описывает архитектурный стиль, уровни, границы модулей, правила зависимостей,
расширения и версионирования. Обоснование каждого решения — в соответствующих ADR.

## 1. Архитектурные принципы (сводка)

Полное описание — `docs/architecture/principles.md`. Кратко: SOLID, DRY, KISS, YAGNI,
Clean Architecture, Domain-Driven Design, Hexagonal (по ADR-002), CQRS (точечно, по ADR-002),
Event-Driven (по ADR-008), API First, Security by Design, Observability by Design.

## 2. Стиль архитектуры

**Микросервисная архитектура, событийно-ориентированная интеграция** (ADR-002, ADR-008).

- Каждый сервис — самостоятельный **ограниченный контекст (bounded context)** с
  собственной схемой данных (**database-per-service**). Прямой доступ к чужой БД запрещён.
- Синхронное взаимодействие — только через публичные REST-контракты (API First, ADR-002).
- Междоменная интеграция — асинхронно через брокер **Apache Kafka** (ADR-008).
- Единственная внешняя точка входа — **API Gateway** (аутентификация, маршрутизация,
  ограничение частоты, CORS).
- Ядро домена (incident) применяет **CQRS**; интеграционные адаптеры оформляются по
  **гексагональному** принципу (порты/адаптеры), наиболее явно в notification-service.

## 3. Уровни системы

```
┌──────────────────────────────────────────────────────────────┐
│ Клиентский уровень:  АРМ диспетчера (SPA), будущие приложения │
├──────────────────────────────────────────────────────────────┤
│ Периметр:            API Gateway (JWT, CORS, rate limiting)   │
├──────────────────────────────────────────────────────────────┤
│ Прикладной уровень:  доменные микросервисы (bounded contexts) │
│                      incident, dispatch, telephony, gis,      │
│                      auth, audit, notification, realtime, ai  │
├──────────────────────────────────────────────────────────────┤
│ Интеграционный:      Apache Kafka (событийная шина)           │
├──────────────────────────────────────────────────────────────┤
│ Уровень данных:      PostgreSQL/PostGIS (per service), Redis  │
├──────────────────────────────────────────────────────────────┤
│ Платформа:           Keycloak (OIDC), Prometheus/Grafana/Loki │
│                      Docker/Kubernetes/Helm                    │
└──────────────────────────────────────────────────────────────┘
```

Внутри каждого Java-сервиса применяется слоистость Clean Architecture:
`domain` (сущности, инварианты, доменные события) → `application` (сервисы вариантов
использования, команды/запросы) → `adapters` (web/REST, persistence/JPA, messaging/Kafka).
Зависимости направлены внутрь: инфраструктура зависит от домена, не наоборот.

## 4. Границы модулей (bounded contexts)

| Контекст | Ответственность | Данные |
|----------|-----------------|--------|
| auth | Пользователи/роли поверх Keycloak | auth_db + Keycloak |
| incident | Ядро: карточки происшествий, статус, история (CQRS) | incident_db + Redis |
| dispatch | Подразделения, наряды, автодиспетчеризация | dispatch_db |
| telephony | Жизненный цикл вызовов, записи | telephony_db |
| gis | Геокодирование, станции, зоны (PostGIS) | gis_db |
| audit | Централизованный журнал (JSONB) | audit_db |
| notification | Каналы Email/SMS/Push (гексагон) | notification_db |
| realtime | Мост Kafka → WebSocket/STOMP | — (stateless) |
| ai | Классификация, NER, извлечение, транскрибация | — (stateless) |
| gateway | Периметр, маршрутизация | — (stateless) |

Полная карта, зависимости и будущие сервисы — `docs/architecture/system-structure.md`.

## 5. Правила зависимостей

1. Сервис владеет своей схемой; чтение/запись чужих таблиц запрещены.
2. Синхронный вызов другого сервиса — только через его REST-контракт (OpenAPI).
3. Предпочтителен асинхронный обмен событиями через Kafka (слабая связанность).
4. Клиенты обращаются к прикладным сервисам только через Gateway.
5. Внутри сервиса домен не зависит от фреймворков/инфраструктуры (Clean Architecture).
6. Общие соглашения (именование, ошибки, версионирование) — единые (см. Coding Standards).
7. Циклические зависимости между сервисами запрещены; при потребности — событие/новый контракт.

## 6. Правила расширения

- Новый функционал — в границах существующего контекста либо новым сервисом (по ADR/RFC),
  если появляется новый bounded context.
- Новый межсервисный обмен — новый Kafka-топик по соглашению именования (см. AGS/DGS) или
  новая версия REST-контракта. Существующие контракты не ломаются без версии.
- Новый источник данных/технология — только через ADR (Architecture Freeze, см. Constitution).

## 7. Правила версионирования

- **REST API:** версия в пути (`/api/v1/...`). Обратно несовместимые изменения — новая
  мажорная версия; старая поддерживается до согласованного вывода из эксплуатации (AGS).
- **События Kafka:** аддитивная эволюция полезной нагрузки; несовместимые изменения — новый
  топик с версией. Консьюмеры толерантны к неизвестным полям.
- **Схема БД:** только через Liquibase-миграции; `ddl-auto: validate`. Обратная совместимость
  на время переходного периода (расширяй-затем-сжимай).
- **Артефакты:** семантическое версионирование сервисов; образы тегируются SHA коммита и
  меткой канала (`stable`).

## 8. Потоки и статусные модели

Событийные потоки (топики Kafka), сценарий «вызов → происшествие → наряд» и статусные
модели (incident/unit/call) описаны в `docs/architecture.md` (реализационная карта) и
`docs/architecture/system-structure.md`. Данный документ фиксирует правила; конкретные
топики и переходы — в реализационной документации, синхронизируемой при изменениях.

## 9. Соответствие Baseline

Любое изменение архитектурного стиля, границ контекстов, правил зависимостей, именования,
структуры каталогов или технологического стека выполняется только через RFC + новый ADR
с Impact Analysis, Migration Plan и Rollback Plan (см. `docs/governance/architecture-baseline.md`).
