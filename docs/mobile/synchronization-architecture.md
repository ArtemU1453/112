# Synchronization Architecture — ЕАСУР

- **Status:** APPROVED · Stage 7 · ADR-017 · Подчиняется Offline Synchronization / Mobile
  Synchronization / Offline Data Governance Standards.

## 1. Модель Offline-First
Клиент работает на локальном шифрованном хранилище (source of truth оффлайн); изменения ставятся в
**outbox-очередь** и синхронизируются при наличии связи. Чтение/запись доступны без сети.

## 2. Двусторонняя дельта-синхронизация (7.5)
- **Pull:** клиент запрашивает изменения с сервера с момента последней синхронизации (курсор/версия).
- **Push:** клиент отправляет накопленные изменения (outbox) пакетно (batch).
- **Дельта:** передаются только изменения (экономия трафика на слабых каналах).
- **Частичная синхронизация:** по областям данных (инцидент/подразделение/формы) и приоритетам.

## 3. Контроль версий и целостность
- Каждая сущность имеет версию (номер/вектор) и контрольную сумму.
- Сервер отклоняет устаревшие записи; клиент дозагружает актуальные.
- Проверка целостности пакета (checksum); повторная передача при сбое (retry).

## 4. Разрешение конфликтов
- База: сравнение версий/времени.
- Доменные правила для авто-разрешения безопасных случаев.
- **Критичные сущности** (статусы, приказы, назначения) — конфликт выносится на **ручное
  разрешение** пользователем (без автозатирания; согласуется с «система не решает за оператора»).
- Все разрешения конфликтов журналируются (журнал синхронизации).

## 5. Восстановление после потери связи
Экспоненциальные повторы; сохранение outbox между сессиями/перезапусками; идемпотентная обработка
на сервере (ключи идемпотентности, Messaging Platform) — отсутствие дублей и потерь.

## 6. Журнал синхронизации
Фиксируются: время, направление (pull/push), объём (дельта), результат, конфликты и их разрешение,
ошибки/повторы. Доступен пользователю и в аудите (Mobile Audit Standard).

## 7. Единый контракт
Синхронизация — через **общий** контракт `contracts/internal/mobile-sync-api.yaml` (batch pull/push,
версии, контрольные суммы), а не app-специфичные API. Транспорт — HTTPS через Gateway; в
air-gapped — локальные endpoint'ы (AGSDS).

## 8. Проверки (обязательные этапа)
Полная работа без сети; устойчивость к обрывам; отсутствие потери данных; журналирование; защита
данных при хранении и передаче — обеспечиваются данной архитектурой и стандартами.
