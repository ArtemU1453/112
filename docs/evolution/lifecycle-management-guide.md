# Lifecycle Management Guide — ЕАСУР

- **Status:** APPROVED · Stage 11 · Реализует Component Lifecycle Standard, Deprecation Policy Standard
  (подэтап 11.1). Практическое руководство по управлению жизненным циклом всех сущностей системы.

## 1. Назначение
Как управлять жизненным циклом компонентов, сервисов, API, инфраструктуры, библиотек, документации и
архитектурных решений: фазы, реестры, владельцы, переходы.

## 2. Классы сущностей и реестры
| Класс | Реестр/источник | Владелец |
|-------|-----------------|----------|
| Сервисы | Service Maturity Model + каталог сервисов | Владелец сервиса |
| API/контракты | `contracts/` + API-каталог (Stage 9) | Владелец контракта |
| Библиотеки/SDK | platform-commons/SDK + каталог версий | Платформенная команда |
| Инфраструктура | IaC (Terraform/Ansible), Helm | Платформенная/Ops |
| Зависимости | Dependency Lifecycle (11.4) | Платформенная команда |
| Документация | docs/ (SSOT), статусы | Владелец раздела |
| Архитектурные решения | ADR/RFC индексы | ARB |

## 3. Фазы (по Component Lifecycle Standard)
`Proposed → Incubating → Active → Maintained → Deprecated → Retired`. Каждая сущность имеет текущий
статус, владельца и дату последнего перехода.

## 4. Переходы (типовой порядок)
1. Инициатива перехода (владелец) с обоснованием.
2. Оценка влияния (совместимость, потребители, безопасность).
3. При затрагивании архитектуры — RFC→ARB→ADR.
4. План (для депрекации/вывода — обязательный план миграции).
5. Исполнение, верификация, фиксация статуса и знаний.

## 5. Правила
- Прекращение поддержки — только с утверждённым планом миграции (Deprecation Policy).
- Документация не удаляется — помечается статусом (Knowledge Governance).
- Устаревание сообщается потребителям (внутр./внешним через реестр интеграций).
