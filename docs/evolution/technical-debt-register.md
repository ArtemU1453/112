# Technical Debt Register — ЕАСУР

- **Status:** APPROVED (реестр, живой документ) · Stage 11 · Реализует Technical Debt Governance
  Standard (подэтап 11.3). Единый реестр технического долга системы.

## 1. Назначение
Регистрация, классификация, оценка, приоритизация и контроль устранения технического долга.
Накопление долга без регистрации запрещено (запрет этапа).

## 2. Схема записи
| Поле | Описание |
|------|----------|
| ID | `TD-YYYY-NNN` (сквозной) |
| Название | Краткая суть |
| Категория | код / архитектура / тесты / инфраструктура / документация / безопасность / данные |
| Источник | PR/инцидент/ревью/аудит |
| Влияние | риск, стоимость обслуживания, влияние на надёжность/безопасность/скорость |
| Приоритет | критический / высокий / средний / низкий |
| План устранения | действие, срок, ответственный, связь с релизом/RFC |
| Статус | open / planned / in-progress / resolved / accepted-risk |

## 3. Реестр
| ID | Название | Категория | Приоритет | Статус | План |
|----|----------|-----------|-----------|--------|------|
| TD-2026-001 | Возможности `CAP-SEARCH-001`/`CAP-DOC-001` в статусе Proposed (движок не выбран) | архитектура | средний | planned | Выбор технологии через RFC+ADR (Technology Roadmap) |
| TD-2026-002 | Пред-существующие замечания линтеров (markdown/yaml) вне контура валидаторов проекта | документация | низкий | open | Плановая гигиена стиля в отдельном PR (не блокирует эксплуатацию) |

> Реестр — живой: новые записи добавляются при внесении компромиссов; критический долг планируется немедленно.

## 4. Процесс
Обнаружение → регистрация → классификация/оценка → приоритизация → план → исполнение → верификация →
закрытие. Тренд долга — метрика (Operational Improvement); отчётность — ARB.

## 5. Правила
- Критический долг (безопасность/надёжность критпути) блокирует накопление, устраняется приоритетно.
- Архитектурный долг устраняется через RFC→ARB→ADR.
- `accepted-risk` допустим только с обоснованием и решением ARB.
