# RFC-0007 — Национальное развёртывание и ввод в эксплуатацию

- **Статус:** Closed (Approved → Implemented → Verified)
- **Автор:** Operational Readiness / Architecture Review (Stage 10)
- **Дата:** 2026-07-21
- **Связанные ADR:** ADR-024 (ввод в эксплуатацию и операционная приёмка)
- **Связанные документы:** Production/Operational/Release Acceptance Standards, National Deployment
  Standard, Go-Live Playbook, итоговые отчёты audit

## 1. Причина изменения
Stage 10 переводит ЕАСУР из состояния «спроектировано и реализовано» в состояние национальной
промышленной эксплуатации. Требуется управляемо зафиксировать процесс ввода (готовность, приёмка,
сертификация, поэтапный go-live, эксплуатация, поддержка) без нарушения Architecture Freeze и без
новой бизнес-функциональности.

## 2. Существующее решение
Stages 1–9 создали архитектуру, платформу, оперативные/мобильные/аналитические/интеграционные
возможности и контракты. Отсутствовали формальные процессы операционной приёмки, сертификации и
поэтапного национального ввода в эксплуатацию.

## 3. Предлагаемое решение
Формальные gate'ы готовности и приёмки, обязательные сертификации (безопасность/производительность/
DR), поэтапное развёртывание (пилот → регион → национальный охват) с критериями продвижения и
откатом, эксплуатационные руководства и долгосрочная поддержка (ADR-024).

## 4. Impact Analysis
- Модули/сервисы: **изменений кода бизнес-логики нет** (запрет этапа); только верификация/оптимизация/
  исправления дефектов при необходимости.
- API/данные: контракты и модель данных не меняются; обратная совместимость сохраняется.
- DevOps: используются существующие CI/CD, IaC, Helm, наблюдаемость, backup/DR (Stages 2–3).
- Безопасность: сертификация без ослабления механизмов (Zero Trust сохранён).

## 5. Compatibility Check
Обратная совместимость полная; изменений контрактов/схем нет; документация синхронизирована;
клиентские приложения не затронуты.

## 6. Риски
Эксплуатационные риски национального ввода; снижаются поэтапностью, gate'ами, планами отката,
сертификациями и обязательным условием отсутствия критических дефектов.

## 7. Migration Plan
Верификация системы → полная программа испытаний → ORR → сертификации (безопасность/произв./DR) →
подготовка эксплуатации → поэтапный go-live (пилот → регион → национальный) → передача в эксплуатацию.

## 8. Rollback Plan
Откат на каждом рубеже go-live по критериям (Go-Live Playbook); откат релиза (`helm rollback`);
восстановление данных (Backup Policy/DR Plan); возврат к предыдущему рубежу без потери контура.
