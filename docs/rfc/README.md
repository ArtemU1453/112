# RFC Process — ЕАСУР

> Приоритет SSOT: 5. RFC (Request for Change) — механизм управляемого изменения архитектуры.
> Статус: APPROVED · Baseline: v1.0

Любое изменение утверждённой архитектуры (стиль, границы контекстов, структура репозитория/
каталогов, правила зависимостей, именование, технологический стек, контракты API, модель
данных) выполняется **только через RFC**, завершающийся новым или обновлённым **ADR**.
Изменение архитектуры «через код» запрещено (Project Constitution, ARTICLE 1, 14).

## Когда требуется RFC

RFC обязателен, если изменение затрагивает хотя бы один пункт Architecture Freeze
(`docs/governance/architecture-baseline.md`) либо ломает обратную совместимость контракта/данных.
Рутинные изменения в границах утверждённой архитектуры (реализация фичи по существующим
контрактам, багфиксы, тесты, документация) RFC **не** требуют.

## Жизненный цикл RFC

```
Proposed → Under Review → Approved → Implemented → Verified → Closed
                       ↘ Rejected
```

Переходы — только последовательные. `Rejected`/`Withdrawn` завершают процесс с фиксацией причины.

## Процедура (обязательная последовательность)

1. Определить **причину** изменения и текущее архитектурное решение.
2. Описать **предлагаемое** решение.
3. Выполнить **Impact Analysis** (модули, сервисы, API, БД, frontend, DevOps, безопасность).
4. Выполнить **Compatibility Check** (обратная совместимость, миграция данных, версия API,
   изменения клиентов, обновление документации).
5. Оценить **риски**.
6. Подготовить **Migration Plan** (подготовка → миграция → тестирование → переключение →
   мониторинг → завершение).
7. Подготовить **Rollback Plan** (критерии, последовательность, восстановление данных/сервисов,
   проверка работоспособности).
8. Оформить RFC по шаблону (`RFC-template.md`), присвоить `RFC-XXXX`.
9. Получить архитектурное решение (Approved).
10. Создать/обновить **ADR** (`ADR-XXXX`).
11. Обновить `Architecture.md`, диаграммы, OpenAPI (если контракты), Database Documentation
    (если модель данных) — до изменения кода.
12. Реализовать изменение; провести Verified; закрыть RFC (Closed).

## Идентификация и хранение

- Имя файла: `RFC-XXXX-kebab-title.md` (сквозная неизменяемая нумерация).
- Каждый RFC содержит уникальный идентификатор, дату, автора, причину, влияние, план миграции,
  статус реализации (Change Control, см. Architecture Baseline).
- Связь RFC ↔ ADR указывается в обоих документах и в ADR Index.

## Document Synchronization

После утверждения изменения обязательно синхронизировать: Vision, Architecture.md, ADR Index,
соответствующий ADR, RFC Index, диаграммы, OpenAPI, Database/ Deployment/ Security/ Testing
Documentation (в затронутой части). Документация не может отставать от реализации
(Constitution, ARTICLE 9, 18).
