# ADR-006 — CI/CD

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** ADR-005, Quality Gates, .gitlab-ci.yml

## Контекст

Нужна автоматизированная сборка, тестирование, упаковка и развёртывание полиглотной системы
с обязательными Quality Gates (тесты, статический анализ, сканирование безопасности) и
управляемым продвижением в staging/production. Требуется совместимость с контуром эксплуатации
и реестром образов.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **GitLab CI/CD** | Единая платформа (репозиторий+CI+реестр+окружения), self-hosted для контура, интеграция с Helm/K8s, ручные gate'ы | Привязка к GitLab |
| GitHub Actions | Популярность, экосистема | Хуже для закрытого контура; отдельный реестр |
| Jenkins | Гибкость, self-hosted | Высокая стоимость сопровождения, groovy-конфигурация |

## Принятое решение

**GitLab CI/CD** (`.gitlab-ci.yml`) со стадиями **build → test → package → deploy**:
- `test`: `mvn verify` для каждого из 9 Java-сервисов (unit + Testcontainers IT),
  `pytest` для ai-service, `npm ci && build/lint` для frontend; отчёты JUnit.
- `package`: сборка и публикация Docker-образов всех 11 контейнеров в реестр (теги SHA + latest),
  только на дефолтной ветке.
- `deploy`: `helm upgrade --install` в staging (авто) и production (manual gate).

## Обоснование

- **Проблема:** воспроизводимый и безопасный конвейер для контура, с обязательными проверками.
- **Почему GitLab CI:** единая платформа (код, CI, реестр контейнеров, окружения) упрощает
  эксплуатацию в закрытом контуре; нативные ручные gate'ы для production; прямая интеграция с
  Helm и Kubernetes.
- **Почему не GitHub Actions/Jenkins:** Actions хуже ложится на закрытый контур и требует
  отдельного реестра; Jenkins дороже в сопровождении.
- **Недостатки/компромиссы:** привязка к GitLab; приемлемо, т.к. платформа покрывает весь цикл.
  При необходимости зеркалирования в GitHub — конвейер остаётся источником истины по CI.
- **Условия пересмотра:** смена платформы репозитория/реестра — через RFC/ADR.

## Последствия

- Quality Gates (см. `docs/standards/quality-gates.md`) реализуются стадиями пайплайна.
- Продвижение в production — только вручную (контроль изменений).
- Интеграционные тесты на Testcontainers выполняются в CI (требуется Docker-in-Docker).
