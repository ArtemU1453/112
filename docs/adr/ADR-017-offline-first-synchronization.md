# ADR-017 — Offline-First и синхронизация данных

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Mobile Architecture, Synchronization Architecture, Offline Synchronization Standard,
  AGSDS, Capability `CAP-SYNC-001`

## Контекст

Полевые подразделения работают в условиях нестабильной/отсутствующей связи. Stage 7 требует
**полной работы без сети** с последующей синхронизацией без потери данных, разрешением конфликтов
и контролем версий. Согласуется с суверенным/автономным характером системы (AGSDS).

## Рассматриваемые варианты

| Аспект | Варианты | Выбор |
|--------|----------|-------|
| Локальное хранилище | Зашифрованная встраиваемая БД (SQLite+шифрование), файловое | **Зашифрованная встраиваемая БД** |
| Модель синхронизации | Двусторонняя дельта-синхронизация с очередью и версиями vs полная перезагрузка | **Дельта-синхронизация + outbox-очередь** |
| Разрешение конфликтов | LWW по версии/времени + доменные правила; ручное разрешение для критичных | **Версионирование + правила + ручное для критичных** |

## Принятое решение

**Offline-First** (`CAP-SYNC-001`): приложения полностью функциональны без сети, данные хранятся
локально в **зашифрованной БД**; изменения ставятся в **очередь синхронизации (outbox)** и
передаются при восстановлении связи. **Двусторонняя дельта-синхронизация** с **контролем версий**
(векторы/номера версий) и **проверкой целостности** (контрольные суммы). **Разрешение конфликтов**:
по версии/времени с доменными правилами; для критичных сущностей — обязательное подтверждение
пользователем (не автоматическое затирание). Ведётся **журнал синхронизации**.

## Обоснование

- **Проблема:** надёжная автономная работа и синхронизация без потери данных при обрывах связи.
- **Почему offline-first + дельта + outbox:** гарантирует работу без сети и устойчивую
  дозагрузку; дельта экономит трафик (важно для слабых каналов); outbox обеспечивает повторную
  передачу и отсутствие потерь.
- **Почему версионирование конфликтов:** предотвращает потерю данных; критичные конфликты решает
  человек (согласуется с политикой «система не решает за оператора»).
- **Компромиссы:** сложность клиентской синхронизации и хранилища; смягчается общим Mobile SDK и
  Offline Data Governance / Mobile Synchronization Standards.
- **Условия пересмотра:** смена стратегии конфликтов/протокола — RFC/ADR.

## Последствия

- Синхронизация — через **общий контракт** (`contracts/internal/mobile-sync-api.yaml`), а не
  app-специфичные API.
- Локальное хранилище шифруется; ключи — в защищённом хранилище устройства (Mobile Security /
  Device Security Standards). `CAP-SYNC-001` в Technology Mapping.
