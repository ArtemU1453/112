# ADR-014 — Абстракция маршрутизации и навигации

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** GIS Architecture, Routing Standard/Governance, TAP, Capability `CAP-ROUTING-001`

## Контекст

Stage 6 требует построения маршрутов, альтернатив, оценки времени прибытия (ETA), расчёта
расстояний с учётом ограничений/перекрытий. Явно **запрещена привязка к конкретному поставщику
навигационных сервисов**. Требуется единый сервис маршрутизации, работоспособный в суверенном
(air-gapped) контуре.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Порт `RoutingProvider` + адаптеры (self-hosted OSRM/Valhalla/GraphHopper)** | Независимость от поставщика (TAP), self-hosted для air-gapped, единый контракт | Требуется эксплуатация движка и данных дорог |
| Прямая интеграция с внешним навигационным API | Быстрый старт | Нарушает запрет привязки; неприемлемо для суверенного контура |
| Собственный маршрутизатор с нуля | Полный контроль | Дорого, дублирует зрелые OSS-движки |

## Принятое решение

**Единый сервис маршрутизации через абстракцию** (`CAP-ROUTING-001`): порт `RoutingProvider`
(построение маршрута, альтернативы, ETA, расстояние, ограничения) с адаптерами к self-hosted
OSS-движкам (например, OSRM/Valhalla/GraphHopper) поверх открытых дорожных данных. Конкретный
движок не фиксируется в прикладном коде; выбор реализации — адаптером и обновлением Technology
Mapping. Все расчёты маршрутов идут **через единый сервис** (обязательная проверка этапа).

## Обоснование

- **Проблема:** нужна маршрутизация без привязки к поставщику и пригодная для air-gapped.
- **Почему абстракция + self-hosted:** соблюдает запрет vendor lock (TAP), обеспечивает
  суверенность (AGSDS), единый контракт для всех потребителей (dispatch/recommendation/map).
- **Почему не внешний API:** прямой запрет привязки; недоступно в изолированном контуре.
- **Компромиссы:** эксплуатация движка и обновление дорожных данных — на стороне платформы;
  смягчается стандартными OSS-решениями и Geospatial Data Governance Standard.
- **Условия пересмотра:** смена движка/данных — обновлением адаптера и Technology Mapping;
  несовместимое изменение контракта маршрутизации — RFC/ADR.

## Последствия

- Контракт `contracts/internal/routing-api.yaml`; порт — в платформенных абстракциях/сервисе GIS.
- `CAP-ROUTING-001` в Technology Mapping; версии движка/данных — каталог версий (TVMS).
- Рекомендации по нарядам (6.6) используют ETA/расстояние единого сервиса, но **не принимают
  решение автоматически** (AI/Decision Support policy).
