# ADR-023 — Безопасность интеграций: mTLS, подпись сообщений, PKI

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Integration Security Architecture, Data Exchange Security Standard, ADR-007, ADR-011,
  ADR-022, Capability `CAP-MSGSIGN-001`

## Контекст

Межведомственный обмен (Stage 9) требует Zero Trust, взаимной аутентификации, авторизации, контроля
целостности, защиты от повторной отправки, подписи сообщений (если применяется архитектурой),
аудита доступа и управления сертификатами. Обмен с внешними организациями — доверие не
предполагается по умолчанию.

## Рассматриваемые варианты

| Аспект | Варианты | Выбор |
|--------|----------|-------|
| Аутентификация каналов | **Взаимный TLS (mTLS) + OAuth2 client credentials** vs только токены | **mTLS + OAuth2** |
| Целостность/неотказуемость сообщений | **Цифровая подпись полезной нагрузки (JWS/detached)** vs только транспорт | **Подпись сообщений** (где применяется) |
| Управление ключами/сертификатами | PKI (ведомственный УЦ) + секрет-менеджер (Vault, ADR-011) | **PKI + Vault** |
| Anti-replay | Nonce/timestamp + идемпотентность | **Nonce/timestamp + idempotency** |

## Принятое решение

**Zero Trust безопасность интеграций** (`CAP-MSGSIGN-001`):
- **Взаимная аутентификация** каналов — **mTLS**; авторизация — OAuth2 client credentials (Keycloak,
  ADR-007) с минимальными привилегиями (scope на интерфейс/операцию).
- **Целостность и неотказуемость** — цифровая **подпись сообщений** (JWS/detached), где применяется
  архитектурой обмена; проверка целостности на приёме.
- **Защита от повторной отправки** — nonce + timestamp + идемпотентность (ключи, Messaging).
- **Управление сертификатами** — ведомственная **PKI**; ключи/секреты — в Vault (ADR-011); ротация.
- **Аудит доступа** — неизменяемый (Immutable Audit; Cross-System Audit Standard).

## Обоснование

- **Проблема:** безопасный межведомственный обмен без доверия по умолчанию, суверенно.
- **Почему mTLS + подпись + PKI:** взаимная аутентификация каналов, целостность/неотказуемость
  сообщений, управляемость сертификатов; соответствует Zero Trust/Least Privilege.
- **Почему не только токены/транспорт:** недостаточно для неотказуемости и межведомственного доверия.
- **Компромиссы:** эксплуатация PKI/подписи; смягчается Vault и стандартизацией (Data Exchange
  Security Standard).
- **Условия пересмотра:** интеграция с национальным УЦ/СМЭВ-аналогом — адаптером и обновлением
  Technology Mapping; иное — RFC/ADR.

## Последствия

- Интеграционная платформа (ADR-022) применяет mTLS/подпись/anti-replay/аудит ко всем внешним обменам.
- Секреты/ключи не в коде (ADR-011). `CAP-MSGSIGN-001` в Technology Mapping.
- Cross-System Audit Standard фиксирует межсистемные обмены неизменяемо.
