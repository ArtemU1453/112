# ADR-022 — Национальная интеграционная платформа

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Integration Architecture (Stage 4), Integration Governance Standard (Stage 4),
  National Integration Architecture, TAP, ADR-008, Capability `CAP-INTEG-PLATFORM-001`

## Контекст

Stage 9 требует единой платформы межведомственного взаимодействия: безопасный, контролируемый,
масштабируемый обмен с внешними системами по REST/gRPC/асинхронно/пакетно/файлово/потоково, с
маршрутизацией и трансформацией сообщений. Принципы: слабая связанность, версионирование
интерфейсов, полная трассируемость, независимость от поставщиков. Запрещены прямые подключения в
обход платформы. Расширяет абстракцию `IntegrationConnector` (Stage 4) до национального масштаба.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Единая интеграционная платформа (integration gateway/ESB) + адаптеры, поверх существующих Gateway/Kafka** | Единая точка обмена, маршрутизация/трансформация, слабая связанность, трассируемость, self-hosted | Отдельный компонент эксплуатации |
| Точечные интеграции сервис-сервис с внешними системами | Быстрее для одной интеграции | Прямые зависимости, нет единого контроля/аудита; запрещено этапом |
| Внешняя облачная iPaaS | Готовые коннекторы | Нарушает суверенность/air-gapped; неприемлемо |

## Принятое решение

**Единая национальная интеграционная платформа** (`CAP-INTEG-PLATFORM-001`), self-hosted:
integration gateway/ESB со **всеми внешними обменами исключительно через платформу** (прямые
подключения запрещены). Поддержка транспортов REST/gRPC/асинхронный(Kafka, ADR-008)/пакетный/
файловый/потоковый через **адаптеры**; маршрутизация и **трансформация** между внутренними и
**каноническими** сообщениями (Canonical Data Model). Внешние системы обязаны быть зарегистрированы
в реестре (9.2). Конкретный ESB-продукт не фиксируется в прикладном коде (TAP; self-hosted, напр.
Apache Camel/маршрутизация поверх Kafka Connect и т. п.).

## Обоснование

- **Проблема:** нужен единый, безопасный, контролируемый, суверенный шлюз межведомственного обмена.
- **Почему единая платформа + адаптеры:** слабая связанность (внешние системы не зависят от
  внутренних), единые аудит/безопасность/мониторинг/версионирование, трансформация в каноническую
  модель; независимость от поставщиков (TAP).
- **Почему не точечные/облако:** прямые зависимости и потеря контроля либо нарушение суверенности.
- **Компромиссы:** эксплуатация ESB-слоя; смягчается переиспользованием Gateway (ADR-007) и Kafka
  (ADR-008) и governance-стандартами.
- **Условия пересмотра:** смена ESB-продукта — адаптером/конфигурацией без изменения контрактов
  (TAP); архитектурное изменение — RFC/ADR.

## Последствия

- Все внешние интеграции — через платформу и реестр (External Systems Registry); прямой обход запрещён.
- Контракты `contracts/external/*` (canonical, versioned); трансформация внутренняя↔каноническая.
- `CAP-INTEG-PLATFORM-001` в Technology Mapping; версии — каталог (TVMS).
