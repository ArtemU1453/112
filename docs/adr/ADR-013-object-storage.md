# ADR-013 — Объектное хранилище

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Backup Policy, Disaster Recovery Plan, Air-Gapped Standard, Storage/Infrastructure
  Architecture, Capability `CAP-STORAGE-001` (ранее Proposed → Approved)

## Контекст

Платформе требуется объектное хранилище для: резервных копий БД (`pg_dump`), артефактов и SBOM,
локального зеркала контейнерных образов/пакетов (air-gapped), а также будущих вложений/записей
(вне текущей Baseline). Требования: self-hosted, суверенное, S3-совместимый API, работа без
внешних облаков.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **S3-совместимое self-hosted (MinIO)** | Стандартный S3 API, self-hosted, HA/erasure coding, простое развёртывание в контуре | Отдельный компонент эксплуатации |
| Ceph (RGW) | Масштабируемость, унификация блок/файл/объект | Высокая сложность эксплуатации |
| NFS/файловое хранилище | Простота | Нет объектной семантики/версионирования/S3 API |
| Облачное объектное хранилище | Управляемость | Неприемлемо для суверенного air-gapped контура |

## Принятое решение

**S3-совместимое объектное хранилище, self-hosted** (реализует `CAP-STORAGE-001`); эталонная
реализация — **MinIO** (HA, erasure coding). Доступ по стандартному **S3 API**, что сохраняет
переносимость (замена реализации без изменения интеграций). Используется для бэкапов, артефактов,
SBOM и локальных зеркал (Air-Gapped Standard). Для крупномасштабных инсталляций допустима замена
на Ceph RGW (тот же S3 API) через обновление Technology Mapping.

## Обоснование

- **Проблема:** нужно суверенное объектное хранилище с S3 API для бэкапов/артефактов/зеркал.
- **Почему S3-совместимое (MinIO):** стандартный API (переносимость и совместимость инструментов
  бэкапа/сканеров), self-hosted, отказоустойчивость (erasure coding), простое развёртывание в контуре.
- **Почему не Ceph по умолчанию:** избыточная эксплуатационная сложность для стартовой инсталляции;
  оставлен как масштабируемая альтернатива с тем же S3 API.
- **Почему не NFS:** отсутствуют объектная семантика, версионирование и S3 API.
- **Недостатки/компромиссы:** ещё один stateful-компонент; смягчается HA-конфигурацией и
  включением в Backup Policy/DR Plan.
- **Условия пересмотра:** рост объёмов/требований → Ceph RGW или ведомственное хранилище (S3 API
  сохраняется) через обновление Technology Mapping; несовместимая замена — RFC/ADR.

## Последствия

- Backup Policy и DR Plan используют объектное хранилище как локальную цель бэкапов.
- Air-Gapped Standard использует его для локальных зеркал артефактов/образов.
- `CAP-STORAGE-001` переведена в Approved (Technology Mapping); версии — в каталоге (TVMS).
