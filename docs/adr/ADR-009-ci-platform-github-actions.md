# ADR-009 — Основная CI-платформа: GitHub Actions (дополняет ADR-006)

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** ADR-006 (CI/CD — GitLab CI), RFC-0001, EAGF, Toolchain Governance Standard,
  Capability `CAP-CI-001`

## Контекст

ADR-006 зафиксировал GitLab CI/CD как конвейер (`.gitlab-ci.yml`). Фактически репозиторий
проекта размещён на GitHub, и рабочий процесс (Pull Request, ревью, автоматизация) ведётся на
GitHub. Этап 2 (Repository Foundation & Development Standards) требует полного набора **GitHub
Actions** workflow'ов (build, test, lint, security, dependency scan, docker build, release,
documentation validation, architecture validation). Возникло расхождение между Baseline (ADR-006)
и требованием Этапа 2.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **GitHub Actions как основная CI + сохранить GitLab CI** | Совпадает с реальным размещением репозитория и процессом PR; выполняет требование Этапа 2; сохраняет переносимость (GitLab CI остаётся зеркалом) | Две конфигурации CI требуют согласованности |
| Заменить GitLab CI на GitHub Actions | Единая платформа | Потеря наработанного `.gitlab-ci.yml`; привязка к одной платформе |
| Оставить только GitLab CI | Соответствие ADR-006 | Не выполняет требование Этапа 2; расходится с фактическим GitHub-процессом |

## Принятое решение

**GitHub Actions — основная CI-платформа проекта.** `.gitlab-ci.yml` **сохраняется** как
поддерживаемый альтернативный/зеркальный конвейер. Данный ADR **дополняет** ADR-006 (не отменяет
его полностью): возможность `CAP-CI-001` теперь реализуется двумя технологиями с приоритетом
GitHub Actions. Обе конфигурации обязаны обеспечивать одинаковые Quality Gates.

## Обоснование

- **Проблема:** Baseline (ADR-006) и фактический процесс/требование Этапа 2 расходятся по CI.
- **Почему GitHub Actions основным:** репозиторий и процесс PR де-факто на GitHub; управление
  ветками, ревью и проверки естественно интегрируются с Actions; Этап 2 требует именно Actions.
- **Почему сохранить GitLab CI:** не терять готовый конвейер и переносимость в закрытый контур
  на базе GitLab (мотивация ADR-006 остаётся валидной для on-premise GitLab).
- **Компромиссы:** поддержка двух конфигураций; смягчается тем, что обе выражают одинаковые
  стадии и Quality Gates, а версии инструментов берутся из единого каталога (TVMS).
- **Влияние на развитие:** новые проверки добавляются в обе конфигурации согласованно; при
  выборе единственной платформы в будущем — отдельный RFC/ADR.
- **Условия пересмотра:** решение о переносе исключительно в GitLab-контур или полном отказе от
  одной из платформ — через новый RFC/ADR.

## Последствия

- Создан полный набор GitHub Actions workflow'ов (`.github/workflows/`).
- `CAP-CI-001` в Technology Mapping ссылается на ADR-006 **и** ADR-009.
- Quality Gates (`docs/standards/quality-gates.md`) реализуются на обеих платформах.
- Изменение оформлено через RFC-0001 с обновлением ADR Index и Technology Mapping (governance-процесс).
