# ADR-007 — Система авторизации

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** ADR-004 (frontend OIDC), Security by Design, gateway-service, auth-service

## Контекст

Государственная система требует централизованной аутентификации, ролевого доступа (RBAC),
единого входа, безопасного управления учётными записями и полного аудита. Клиент (SPA) и
множество backend-сервисов должны единообразно проверять личность и права без хранения
паролей в приложениях.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Keycloak (OIDC/OAuth2)** | Готовый IdP, realm/roles/clients, self-hosted, стандарты OIDC/JWT, админ-API | Требует эксплуатации отдельного компонента |
| Собственная auth (JWT вручную) | Полный контроль | Дорого и рискованно: своя криптография, ротация ключей, аудит — антипаттерн для госсистемы |
| Внешний облачный IdP | Быстрый старт | Неприемлемо для закрытого контура и суверенности данных |

## Принятое решение

**Keycloak** как поставщик идентичности (realm `emergency-112`):
- **OIDC/OAuth2**, JWT-токены; frontend — **Authorization Code + PKCE (S256)**.
- **RBAC-роли:** `ROLE_DISPATCHER`, `ROLE_SENIOR_DISPATCHER`, `ROLE_ADMIN`, `ROLE_CREW`,
  `ROLE_ANALYST`; извлекаются из claim `realm_access.roles` и маппятся в Spring authorities.
- **Gateway** валидирует токен (issuer Keycloak) на периметре; каждый сервис — OAuth2 resource
  server.
- **auth-service** управляет пользователями/ролями через Keycloak Admin API и публикует
  события в аудит.

## Обоснование

- **Проблема:** нужна централизованная, стандартная и аудируемая аутентификация/авторизация
  для SPA и множества сервисов.
- **Почему Keycloak:** зрелый self-hosted IdP со стандартами OIDC/JWT, ролями и админ-API;
  снимает необходимость собственной криптографии; разворачивается в контуре.
- **Почему не собственная реализация:** самостоятельная работа с токенами/ключами/ротацией —
  источник уязвимостей; недопустимо для госсистемы (Security by Design).
- **Почему не облачный IdP:** нарушает требования закрытого контура и суверенности.
- **Недостатки/компромиссы:** дополнительный эксплуатируемый компонент и точка отказа —
  компенсируется резервированием Keycloak и стандартностью решения.
- **Условия пересмотра:** интеграция с ведомственной СУДИР/федерацией — через RFC/ADR
  (Keycloak поддерживает федерацию, вероятно эволюция, а не замена).

## Последствия

- Секреты клиентов/админа — вне git, из защищённого хранилища.
- Роли и учётные записи-заглушки realm — только для dev/stage (см. operations.md).
- Любое изменение ролевой модели фиксируется в Security Documentation и, при влиянии на
  контракты, — новым ADR.
