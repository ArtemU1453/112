# ADR-008 — Брокер сообщений

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** ADR-002 (событийная интеграция), Architecture.md, System Structure

## Контекст

Микросервисы интегрируются асинхронно (ADR-002). Требуется надёжная шина событий с
сохранением сообщений, воспроизведением, разделением на партиции для параллелизма,
устойчивостью к всплескам нагрузки (шторм вызовов) и поддержкой нескольких независимых
потребителей одного события (fan-out): incident, dispatch, notification, realtime, audit.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Apache Kafka** | Персистентный лог, воспроизведение, партиции, высокая пропускная способность, множественные consumer-группы (fan-out), порядок в пределах ключа | Сложнее в эксплуатации |
| RabbitMQ | Простые очереди, гибкая маршрутизация | Слабее для воспроизведения/лога событий и высокого fan-out по истории |
| Redis Streams | Лёгкий, уже есть Redis | Меньшие гарантии/зрелость для критичной событийной шины |
| БД как очередь (outbox-only) | Без нового компонента | Не масштабируется как шина; сложный fan-out |

## Принятое решение

**Apache Kafka** как событийная шина. Топики и потоки:

| Топик | Публикует | Потребляют |
|-------|-----------|------------|
| `incident.created` | incident | dispatch, realtime |
| `incident.updated` | incident | realtime |
| `dispatch.assigned` | dispatch | incident, notification, realtime |
| `unit.status-changed` | dispatch | realtime |
| `call.received` | telephony | ai |
| `call.analyzed` | ai | telephony, incident, realtime |
| `notification.requested` | любой сервис | notification |
| `audit.events` | все сервисы | audit |

Консьюмеры **идемпотентны**, толерантны к повторам и неизвестным полям; продюсеры —
с гарантиями доставки (идемпотентность/acks). realtime-service мультиплексирует Kafka в STOMP,
чтобы клиентские WebSocket-соединения не потребляли партиции напрямую.

## Обоснование

- **Проблема:** нужна устойчивая шина с историей, партиционированием и fan-out для 112.
- **Почему Kafka:** персистентный лог позволяет воспроизведение и подключение новых
  потребителей; партиции дают параллелизм и порядок по ключу; выдерживает всплески (шторм
  вызовов); множественные consumer-группы обслуживают fan-out на incident/dispatch/notification/
  realtime/audit из одного события.
- **Почему не RabbitMQ:** ориентирован на очереди/маршрутизацию, слабее для лог-ориентированного
  воспроизведения и историко-ориентированного fan-out.
- **Почему не Redis Streams/БД-очередь:** недостаточные зрелость/гарантии для критичной шины
  либо отсутствие масштабируемого fan-out.
- **Недостатки/компромиссы:** эксплуатационная сложность Kafka и eventual consistency; приняты
  осознанно; смягчаются идемпотентностью, мониторингом лага и стандартами именования (DGS/AGS).
- **Влияние на развитие:** новые интеграции — через новый топик по соглашению именования или
  новую версию контракта; существующие payload'ы эволюционируют аддитивно.
- **Условия пересмотра:** если профиль нагрузки окажется низким и fan-out тривиальным —
  упрощение шины через RFC; маловероятно для 112.

## Последствия

- Обязателен мониторинг consumer lag (см. operations.md).
- Схемы событий версионируются; несовместимые изменения — новый топик с версией.
- Порядок гарантируется только в пределах ключа партиции — учитывается в дизайне консьюмеров.
