# ADR-011 — Управление секретами

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** ADR-007 (авторизация), Security Architecture, Environment Governance Standard,
  Air-Gapped Standard, Capability `CAP-SECRETS-001` (ранее Proposed → Approved)

## Контекст

Секреты (учётные данные БД, клиентов Keycloak, ключи, токены) не хранятся в git/образах/открытых
конфигурациях (Constitution ARTICLE 10, RGS, EGS). Требуется централизованное хранилище с
контролем доступа, аудитом, ротацией и аварийным восстановлением, работоспособное в изолированном
контуре. Приложениям секреты должны доставляться как Kubernetes Secret без хранения значений в git.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **HashiCorp Vault + External Secrets Operator (ESO)** | Централизованное хранилище, динамические секреты, аудит, ротация, self-hosted/air-gapped; ESO синхронизирует в K8s Secret | Отдельный компонент для эксплуатации |
| Sealed Secrets | Простота, git-совместимость (шифрованные секреты в git) | Нет ротации/аудита/динамических секретов; хуже для гос-требований |
| Только Kubernetes Secrets | Встроено | Base64, не шифрование; нет ротации/аудита/централизованного контроля |
| Облачный секрет-менеджер | Управляемость | Неприемлемо для суверенного air-gapped контура |

## Принятое решение

**HashiCorp Vault** как централизованное хранилище секретов (реализует `CAP-SECRETS-001`),
**External Secrets Operator** для синхронизации секретов в Kubernetes (`ExternalSecret` →
`Secret`). Аутентификация рабочих нагрузок в Vault — через Kubernetes auth (ServiceAccount).
Развёртывание Vault — self-hosted внутри контура (HA-режим), без внешних зависимостей.

## Обоснование

- **Проблема:** нужен суверенный, аудируемый, ротируемый источник секретов для K8s-нагрузок.
- **Почему Vault + ESO:** централизованное шифрованное хранилище с политиками доступа, аудит-логом
  и ротацией; ESO даёт декларативную доставку секретов в кластер без значений в git; всё self-hosted,
  пригодно для air-gapped.
- **Почему не Sealed Secrets:** отсутствуют ротация, аудит и динамические секреты — недостаточно
  для государственных требований ИБ.
- **Почему не «голые» Secrets:** base64 ≠ защита; нет централизованного контроля/аудита.
- **Недостатки/компромиссы:** Vault — критичный компонент с собственной эксплуатацией (unseal,
  HA, бэкап). Смягчается HA-развёртыванием, процедурами unseal и DR (Disaster Recovery Plan).
- **Условия пересмотра:** появление ведомственного HSM/KMS → интеграция как auto-unseal/backend
  через RFC/ADR.

## Последствия

- Структура доступа, политика ротации, аудит доступа и аварийное восстановление — в Secrets
  Management (`docs/platform/secrets-management.md`) и Security Architecture.
- Манифесты `ExternalSecret`/`SecretStore` — в `kubernetes/platform` (без реальных значений).
- `CAP-SECRETS-001` переведена в статус Approved (Technology Mapping); версии — в каталоге.
