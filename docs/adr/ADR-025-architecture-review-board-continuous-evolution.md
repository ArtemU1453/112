# ADR-025 — Architecture Review Board и непрерывная эволюция

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Architecture Evolution Standard, Continuous Compliance Standard, EAGF, Project
  Constitution, Architecture Baseline, RFC-процесс, ADR-024, Enterprise Evolution Strategy

## Контекст

Система введена в промышленную эксплуатацию (Stage 10, ADR-024) и должна эксплуатироваться и
развиваться десятилетиями. Требуется зафиксировать **орган и процесс** управляемой эволюции, чтобы
изменения архитектуры не проводились «через код», технический долг не накапливался бесконтрольно, а
соответствие архитектуре проверялось непрерывно. Ранее (Stage 1–2) RFC завершался «архитектурным
решением»; на долгосрочном горизонте нужен формализованный **Architecture Review Board (ARB)**.

## Рассматриваемые варианты

| Аспект | Варианты | Выбор |
|--------|----------|-------|
| Орган решений | Ad-hoc ревью vs **постоянный Architecture Review Board (ARB)** | **ARB** |
| Путь изменения | Прямые правки vs **RFC → ARB → ADR** | **RFC → ARB → ADR** |
| Контроль соответствия | Разовый аудит vs **непрерывный (drift/compliance) + периодические ревью** | **Непрерывный + периодический** |
| Технический долг | Неформально vs **реестр с классификацией/приоритизацией/планом** | **Реестр (Technical Debt Register)** |

## Принятое решение

**Непрерывная управляемая эволюция под контролем ARB:**
- Любое изменение утверждённой архитектуры — **только** по маршруту **RFC → Architecture Review Board
  → ADR** (изменение архитектуры «через код» запрещено; Constitution, ARTICLE 1/14).
- **ARB** — постоянный орган: рассматривает RFC, ведёт периодический архитектурный аудит (drift,
  соответствие ADR/RFC/принципам EA), утверждает модернизацию, депрекацию и внедрение технологий.
- **Непрерывный контроль соответствия** (Continuous Compliance): автоматические валидаторы (архитектура/
  версии/документация) в CI + регулярные ревью ARB.
- **Технический долг** регистрируется, классифицируется, оценивается и приоритизируется (Technical
  Debt Register); накопление без регистрации запрещено.
- **Жизненный цикл** компонентов/сервисов/API/зависимостей/инфраструктуры/документации управляется
  (Component/Dependency Lifecycle, Deprecation Policy); прекращение поддержки — только с планом миграции.

## Обоснование

- **Проблема:** сохранить архитектурную целостность и суверенность на горизонте 10+ лет при
  постоянных изменениях, обновлениях и модернизации.
- **Почему ARB + RFC→ADR:** единая точка архитектурной ответственности, прослеживаемость решений,
  предотвращение дрейфа; соответствует EAGF/Constitution.
- **Почему непрерывный контроль:** дрейф и долг обнаруживаются рано, а не при аварии.
- **Компромиссы:** процессные издержки; оправданы критичностью национальной системы.
- **Условия пересмотра:** изменение состава/полномочий ARB или маршрута изменений — через RFC/ADR.

## Последствия

- Все последующие ADR/RFC проходят ARB; периодические ревью фиксируются (аудит, EAGF).
- Технический долг, депрекации и модернизация ведутся по стандартам Stage 11.
- Enterprise Evolution Strategy/Roadmap и Technology Roadmap становятся управляющими документами развития.
