# ADR-010 — Инструменты Infrastructure as Code

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** ADR-005 (контейнеризация/оркестрация), Platform Architecture, Air-Gapped Standard,
  Capability `CAP-IAC-001`

## Контекст

Платформа ЕАСУР разворачивается воспроизводимо в защищённом (в т. ч. air-gapped) контуре.
Требуется декларативное описание инфраструктуры: платформенный слой Kubernetes (namespaces,
RBAC, квоты, сетевые политики, storage classes) и подготовка узлов/ОС и локальных сервисов
(реестр образов, зеркала). Целевая среда — on-premise, без привязки к публичному облаку.

## Рассматриваемые варианты

| Потребность | Варианты | Выбор |
|-------------|----------|-------|
| Декларативная инфраструктура (кластерные ресурсы, storage, network) | Terraform, Pulumi, Crossplane, «голые» манифесты | **Terraform** (провайдеры `kubernetes`, `helm`) |
| Подготовка узлов/ОС, локальные сервисы, обновления | Ansible, Salt, скрипты | **Ansible** |
| Наложение окружений на манифесты | Kustomize, Helm | **Kustomize** (ADR-005) + Helm для приложений |

## Принятое решение

**Terraform + Ansible** как инструменты IaC (реализуют `CAP-IAC-001`):
- **Terraform** — декларативное управление платформенным слоем через провайдеры `kubernetes` и
  `helm` (namespaces, RBAC, resource quotas, limit ranges, network policies, storage classes,
  установка платформенных чартов). Провайдер-агностично к облаку: управляет ресурсами кластера,
  а не конкретного публичного провайдера.
- **Ansible** — подготовка узлов (ОС, контейнерный рантайм, узлы Kubernetes), развёртывание
  локального реестра образов и зеркал, процедуры офлайн-обновления (Air-Gapped Standard).
- **Kustomize** (ADR-005) — окружение-специфичные оверлеи манифестов; **Helm** — приложения.

## Обоснование

- **Проблема:** нужна воспроизводимая, версионируемая инфраструктура для on-prem/air-gapped.
- **Почему Terraform:** зрелый декларативный инструмент с состоянием и планом изменений;
  провайдеры `kubernetes`/`helm` дают конкретное управление платформой без облачной привязки;
  переносим в изолированный контур (провайдеры кэшируются локально).
- **Почему Ansible:** агентless, идемпотентные плейбуки для подготовки узлов/ОС и офлайн-операций;
  естественен для on-prem и мобильных/резервных площадок.
- **Почему не Pulumi/Crossplane:** Pulumi тянет языковой рантайм; Crossplane требует управляющего
  кластера и внешних провайдеров — избыточно и хуже для air-gapped.
- **Недостатки/компромиссы:** два инструмента (Terraform+Ansible) с разными моделями; принято
  ради разделения ответственности (кластерные ресурсы vs подготовка узлов). Состояние Terraform
  хранится в защищённом backend внутри контура.
- **Условия пересмотра:** переход в managed-Kubernetes публичного облака → пересмотр провайдеров
  через RFC/ADR.

## Последствия

- Структура `infrastructure/terraform` (модули + окружения) и `infrastructure/ansible` (роли/плейбуки).
- Версии инструментов — в каталоге версий (`config/toolchain/versions.yaml`, TVMS).
- `CAP-IAC-001` добавлена в Technology Mapping.
