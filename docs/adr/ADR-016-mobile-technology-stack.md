# ADR-016 — Технологический стек мобильной экосистемы

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Mobile Architecture, Mobile SDK Guide, ADR-004 (frontend), TAP, Capability `CAP-MOBILE-001`

## Контекст

Stage 7 требует единой мобильной экосистемы для нескольких ролей (подразделения, начальники
караулов, руководители ликвидации ЧС и др.). Все приложения обязаны использовать **единую
платформу, единый SDK, единые API и политики безопасности**. Требования: offline-first, работа на
Android (основная целевая ОС полевых устройств), безопасность (шифрование хранилища, защищённые
ключи), суверенность (распространение вне публичных сторов — MDM/внутренний канал).

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Единый кросс-платформенный стек (Flutter или React Native) + общий Mobile SDK** | Один код на несколько приложений/платформ, единый SDK/дизайн-система, быстрая разработка, offline-библиотеки | Требует дисциплины единых компонентов |
| Нативные приложения на каждую платформу отдельно | Максимальная нативность | Дублирование, дороже, сложнее держать единые SDK/дизайн |
| Мобильный веб (PWA) | Переиспользование web-стека (ADR-004) | Слабее offline/аппаратная безопасность/фон; неприемлемо для полевых условий |

## Принятое решение

**Единый кросс-платформенный мобильный стек** (реализует `CAP-MOBILE-001`) с **общим Mobile SDK**
(`libraries/mobile-*`) и единой дизайн-системой. Конкретный фреймворк (Flutter либо React Native)
фиксируется реализацией и каталогом версий (TVMS); архитектурно важно, что стек **единый для всех
приложений** и не привязывает бизнес-логику к фреймворку (TAP). Целевая ОС — Android (полевые
устройства); iOS — при необходимости тем же кодом. Приложения: Field Responder (`apps/mobile-crew`),
Commander (`apps/mobile-commander`), Incident Command (`apps/mobile-command`).

## Обоснование

- **Проблема:** несколько мобильных приложений с общими требованиями (SDK/API/безопасность/offline).
- **Почему единый кросс-платформенный стек:** исключает дублирование, обеспечивает единый SDK и
  дизайн-систему (требование этапа), ускоряет разработку, покрывает offline-first библиотеками.
- **Почему не PWA:** недостаточные offline/фоновые/аппаратно-защитные возможности для полевых
  условий; MDM-дистрибуция сложнее.
- **Компромиссы:** дисциплина единых компонентов и SDK; смягчается Internal SDK Governance /
  Mobile SDK Guide и дизайн-системой.
- **Условия пересмотра:** смена фреймворка — обновлением реализации/каталога версий без изменения
  контрактов и SDK-абстракций; архитектурное изменение — RFC/ADR.

## Последствия

- Единый Mobile SDK и дизайн-система обязательны для всех приложений (Definition of Done этапа).
- Мобильные приложения используют **общие платформенные API** (запрет app-специфичных API без ADR).
- `CAP-MOBILE-001` в Technology Mapping; версии — каталог версий (TVMS).
