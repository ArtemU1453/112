# ADR-015 — Абстракция поставщиков картографических данных и телеметрии

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** GIS Architecture, Map Layer Governance / Telemetry Abstraction Standards, TAP,
  Capabilities `CAP-MAP-001`, `CAP-TELEMETRY-001`

## Контекст

Stage 6 явно **запрещает привязку к конкретному поставщику картографических данных**, а
отслеживание ресурсов (6.8) должно работать с **различными источниками телеметрии через адаптеры**.
Требуется суверенность (air-gapped): карта и телеметрия не зависят от внешних облачных сервисов.

## Рассматриваемые варианты

| Область | Варианты | Выбор |
|---------|----------|-------|
| Картографические данные/тайлы | Порт `MapDataProvider` + self-hosted тайлы (OSM-данные, MBTiles/vector) vs внешний тайл-сервис | **Абстракция + self-hosted** |
| Телеметрия местоположения | Порт `TelemetrySource` + адаптеры (GPS/АРМ экипажа/внешние трекеры) vs жёсткая интеграция | **Абстракция + адаптеры** |

## Принятое решение

- **Картография (`CAP-MAP-001`):** порт `MapDataProvider` (базовые/тематические слои, тайлы,
  геокодирование-данные) с self-hosted реализацией на открытых данных (OSM-совместимые, локальные
  тайлы). Конкретный поставщик/формат не фиксируется в прикладном коде.
- **Телеметрия (`CAP-TELEMETRY-001`):** порт `TelemetrySource` (позиция/движение/статусные события
  техники) с адаптерами к разным источникам; прикладная логика отделена от телеметрии
  (обязательная проверка этапа). Frontend-карта использует данные через абстракции (OpenLayers,
  ADR-004, поверх self-hosted слоёв).

## Обоснование

- **Проблема:** запрет vendor lock для карт и требование мультиисточниковой телеметрии.
- **Почему абстракции + self-hosted:** соблюдение TAP и суверенности (AGSDS); заменяемость
  поставщика/источника без изменения потребителей; единые слои для оперативной карты.
- **Почему не внешние сервисы:** прямой запрет и недоступность в air-gapped.
- **Компромиссы:** хранение/обновление гео-данных и эксплуатация тайл-сервера — на платформе
  (Geospatial Data Governance Standard); телеметрия разнородна — унифицируется портом.
- **Условия пересмотра:** смена формата тайлов/источника телеметрии — адаптером и Technology
  Mapping; несовместимость — RFC/ADR.

## Последствия

- Контракты `resource-tracking-api.yaml` (телеметрия), `operational-map-api.yaml` (слои/карта).
- `CAP-MAP-001`, `CAP-TELEMETRY-001` в Technology Mapping; версии данных/тайлов — каталог (TVMS).
- Историю перемещений запрещено удалять (append-only); координаты объектов меняются только с
  фиксацией (Spatial Security / Geospatial Data Governance Standards).
