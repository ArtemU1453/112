# ADR-019 — Аналитическое хранилище данных (Data Warehouse)

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Data Warehouse Architecture, Analytics Architecture, ADR-003 (БД), ADR-008 (Kafka),
  TAP, Capability `CAP-DWH-001`

## Контекст

Stage 8 требует единого аналитического пространства: исторические данные, агрегаты, витрины,
каталог данных, контроль качества. Оперативные БД сервисов (database-per-service, ADR-003) не
предназначены для тяжёлой аналитики (нагрузка на оперативный контур). Требуется отделить
аналитику от оперативного реагирования, обеспечить воспроизводимость и прослеживаемость (lineage),
суверенность (self-hosted, air-gapped).

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Отдельное аналитическое хранилище (DWH) с ETL/ELT из событий Kafka и снапшотов БД** | Отделение от оперативного контура, исторические данные/витрины, воспроизводимость, self-hosted | Дублирование данных, ETL-конвейеры |
| Аналитика поверх оперативных БД напрямую | Нет дублирования | Нагрузка на оперативный контур, риск для реагирования; плохо для истории/агрегатов |
| Внешнее облачное DWH | Управляемость | Нарушает суверенность/air-gapped; неприемлемо |

## Принятое решение

**Отдельное аналитическое хранилище данных (DWH)** (`CAP-DWH-001`), self-hosted, наполняемое из
**событий Kafka** (ADR-008) и периодических снапшотов оперативных БД через ETL/ELT. Реализуется
единым `analytics-service` (`services/analytics-service`) — **без дублирования аналитических
платформ** (запрет этапа). Технология DWH не фиксируется в прикладном коде (PostgreSQL как
аналитическая БД/колоночное расширение или совместимое OLAP-решение; TAP). Обеспечивает
исторические данные, агрегаты, витрины, каталог данных, контроль качества и **прослеживаемость
происхождения** (lineage).

## Обоснование

- **Проблема:** нужна аналитика без нагрузки на оперативный контур, с историей/агрегатами/lineage.
- **Почему отдельное DWH из событий/снапшотов:** изоляция аналитики (не влияет на латентность
  реагирования, R-P1); события Kafka дают потоковую загрузку и восстановимость; воспроизводимость
  отчётов (фиксированные срезы).
- **Почему не поверх оперативных БД:** риск для оперативного контура и плохая история/агрегация.
- **Почему self-hosted:** суверенность и air-gapped (AGSDS).
- **Компромиссы:** дублирование данных и ETL-конвейеры; смягчается Data Lineage / Data Quality /
  Master Data Governance стандартами.
- **Условия пересмотра:** рост объёмов → колоночное/распределённое OLAP-решение (тот же контракт);
  несовместимое изменение — RFC/ADR.

## Последствия

- Единый `analytics-service` (реализация DWH/витрин/отчётов/прогнозов) — не дублируется.
- **Исторические данные неизменяемы**; изменения только с полным аудитом (запрет этапа).
- `CAP-DWH-001` в Technology Mapping; версии — каталог (TVMS). Retention — Data Retention for
  Analytics Standard.
