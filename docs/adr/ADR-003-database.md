# ADR-003 — База данных

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** ADR-002 (database-per-service), Database Governance Standard

## Контекст

Требуется транзакционное реляционное хранилище с сильной согласованностью для доменных
данных, поддержкой геопространственных запросов (координаты, расстояния, зоны ответственности),
гибким хранением разнородных событий аудита и кэшированием оперативных чтений. Каждый сервис
владеет своей схемой (ADR-002).

## Рассматриваемые варианты

| Потребность | Варианты | Выбор |
|-------------|----------|-------|
| Транзакционное реляционное хранилище | PostgreSQL, MySQL/MariaDB, Oracle | **PostgreSQL 16/17** |
| Геоданные | PostGIS, отдельная geo-БД, вычисления в коде | **PostGIS** (расширение PostgreSQL) |
| Гибкие payload'ы аудита | JSONB в PostgreSQL, отдельная документная БД | **JSONB (PostgreSQL)** |
| Кэш/rate limiting | Redis, Memcached, Hazelcast | **Redis** |

## Принятое решение

- Основное хранилище — **PostgreSQL** (по одной БД на сервис).
- Геопространственные функции — **PostGIS** (в gis-service и для координат): `ST_Distance`,
  `ST_Contains`, KNN-оператор `<->`, `pg_trgm` для нечёткого поиска адресов.
- Разнородные события аудита — **JSONB** с GIN-индексом (audit-service).
- **Redis** — кэш статистики происшествий (короткий TTL) и backend rate limiting шлюза.
- Версионирование схемы — **Liquibase**, `ddl-auto: validate`.

## Обоснование

- **Проблема:** нужны ACID, геозапросы, гибкий аудит и кэш — по возможности без зоопарка СУБД.
- **Почему PostgreSQL:** ACID, зрелость, открытая лицензия, PostGIS и JSONB в одной СУБД
  сокращают число технологий; сильная экосистема резервного копирования и мониторинга.
- **Почему не Oracle:** лицензионная стоимость и зависимость от вендора без функциональной
  необходимости для проекта.
- **Почему PostGIS, а не расчёты в коде:** индексируемые геозапросы (KNN, содержание в зоне)
  на порядки эффективнее и точнее ручных вычислений при росте данных; в коде оставлен только
  быстрый Haversine для авто-подбора наряда.
- **Почему JSONB для аудита:** события разнородны по структуре; JSONB + GIN даёт гибкость и
  поиск без жёсткой схемы на каждый тип события.
- **Почему Redis:** быстрый кэш и распределённый счётчик для rate limiting; альтернативы не
  дают преимуществ в этом контексте.
- **Недостатки/компромиссы:** database-per-service усложняет кросс-доменные выборки (решается
  событиями и, при необходимости, отдельным analytics-сервисом); JSONB слабее типизирован —
  ограничивается только аудитом, не доменными агрегатами.
- **Условия пересмотра:** экстремальный рост объёмов аудита → выделенное хранилище/партиционирование
  (RFC); потребность в аналитике → отдельная витрина.

## Последствия

- Инициализация БД и расширений — `infrastructure/postgres/init`.
- Все изменения схемы — только миграциями Liquibase (Database First, Constitution ARTICLE 13).
