# ADR-020 — Платформа BI и визуализации

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** BI Architecture, Executive Dashboard / Dashboard Design / BI Security Standards, ADR-019,
  TAP, Capability `CAP-BI-001`

## Контекст

Stage 8 требует дашбордов для всех уровней управления (диспетчер … центральный аппарат) с
показателями в реальном времени и историей, drill-down, фильтрацией, экспортом и национальным
ситуационным центром. Требуется self-hosted визуализация (суверенность, air-gapped), единая для
всех уровней (без дублирующих BI-платформ).

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Self-hosted BI/визуализация (Grafana/Metabase/Superset) поверх DWH + API дашбордов** | Готовые дашборды/drill-down/экспорт, self-hosted, интеграция с Prometheus/Loki (Stage 3) | Ещё один компонент эксплуатации |
| Собственная BI с нуля | Полный контроль | Дорого, дублирует зрелые OSS |
| Внешний облачный BI | Быстрый старт | Нарушает суверенность; неприемлемо |

## Принятое решение

**Self-hosted BI/визуализация** (`CAP-BI-001`) поверх аналитического хранилища (ADR-019).
Оперативные дашборды реального времени — интеграция с уже используемым Grafana (Stage 3, поверх
Prometheus/Loki) и/или BI-инструментом (Metabase/Superset) для управленческой аналитики.
Конкретный инструмент не фиксируется в прикладном коде (TAP); данные дашбордов доступны и через
**API дашбордов** (`contracts`), чтобы веб-АРМ/ситуационный центр могли рендерить единообразно.
Единая BI-платформа для всех уровней — **без дублирующих аналитических платформ** (запрет этапа).

## Обоснование

- **Проблема:** дашборды для многих ролей с real-time/history/drill-down/экспортом, суверенно.
- **Почему self-hosted OSS BI:** готовые возможности, интеграция с существующей наблюдаемостью
  (Grafana), self-hosted (AGSDS); API дашбордов обеспечивает единообразный рендер в АРМ/ситуационном
  центре.
- **Почему не облако/своё с нуля:** суверенность/стоимость.
- **Компромиссы:** эксплуатация BI-компонента и разграничение доступа (BI Security Standard);
  смягчается RBAC и разделением уровней управления.
- **Условия пересмотра:** смена BI-инструмента — без изменения контрактов дашбордов (TAP); иное — RFC/ADR.

## Последствия

- Дашборды по уровням управления (Executive Dashboard Standard); доступ по RBAC (BI Security).
- Национальный ситуационный центр — поверх той же BI/DWH (National Situation Center Architecture).
- `CAP-BI-001` в Technology Mapping; версии — каталог (TVMS).
