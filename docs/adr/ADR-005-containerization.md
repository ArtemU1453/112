# ADR-005 — Контейнеризация и оркестрация

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** ADR-002, ADR-006 (CI/CD), operations.md, helm/emergency-112

## Контекст

Полиглотная система (Java, Python, статический frontend) требует воспроизводимой упаковки,
единообразного развёртывания в нескольких средах (dev/stage/prod), горизонтального
масштабирования, самовосстановления и управляемой конфигурации. Целевая среда — защищённый
контур эксплуатации.

## Рассматриваемые варианты

| Уровень | Варианты | Выбор |
|---------|----------|-------|
| Упаковка | Docker/OCI-образы, нативные пакеты | **Docker/OCI** |
| Локально/stage | Docker Compose, K8s локально | **Docker Compose** |
| Оркестрация (prod) | Kubernetes, Nomad, Swarm | **Kubernetes** |
| Управление релизами | Helm, Kustomize, raw manifests | **Helm (зонтичный чарт)** |

## Принятое решение

- Все сервисы упаковываются в **Docker/OCI-образы** (многостадийные сборки).
- **Docker Compose** — локальная разработка и staging целиком (инфраструктура + сервисы).
- **Kubernetes** — целевая промышленная оркестрация.
- **Helm** — зонтичный чарт `emergency-112`: обобщённые шаблоны Deployment/Service по всем
  сервисам (range по values), ConfigMap/Secret, Ingress, HPA, CronJob резервного копирования;
  прод-оверрайды в `values-production.yaml`.

## Обоснование

- **Проблема:** воспроизводимость и единообразие деплоя полиглотной системы в разных средах.
- **Почему Docker:** стандарт упаковки, изоляция зависимостей, единый артефакт для CI/CD.
- **Почему Compose локально:** минимальный порог входа для разработчика и быстрый end-to-end
  запуск без кластера.
- **Почему Kubernetes:** самовосстановление, горизонтальное масштабирование (HPA), декларативная
  конфигурация, стандарт для гос/enterprise-эксплуатации в контуре.
- **Почему Helm, а не Kustomize/raw:** параметризация и переиспользуемые шаблоны для 11 сервисов
  из одного источника; управление окружениями через values; интеграция с CI (`helm upgrade`).
- **Недостатки/компромиссы:** Kubernetes повышает требования к эксплуатации; Helm-шаблоны
  сложнее в отладке — компенсируется единым обобщённым шаблоном и `helm template` в проверках.
- **Условия пересмотра:** managed-Kubernetes vs on-premise — вопрос инфраструктуры, не архитектуры;
  смена оркестратора — через RFC/ADR.

## Последствия

- Единый обобщённый шаблон снижает дублирование манифестов.
- Инфраструктурные зависимости (PostgreSQL, Kafka, Redis, Keycloak) подключаются как внешние
  endpoint'ы/операторы, а не разворачиваются прикладным чартом.
- Секреты — вне git; в проде передаются из защищённого хранилища (см. operations.md).
