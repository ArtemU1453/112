# ADR-002 — Архитектурный стиль

- **Статус:** Accepted
- **Дата:** Project Start
- **Связи:** Architecture.md, Architecture Principles, ADR-008 (брокер), ADR-003 (БД)

## Контекст

Система объединяет разнородные функции (телефония, карточка происшествия, диспетчеризация,
геоинформация, AI, уведомления, реальное время) с разными профилями нагрузки и разной
скоростью изменения. Требуются независимое развёртывание и масштабирование отдельных
функций, отказоустойчивость и слабая связанность, а также богатая доменная модель ядра.

## Рассматриваемые варианты

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Микросервисы + событийная интеграция (Kafka)** | Независимое масштабирование/деплой, изоляция отказов, границы контекстов, слабая связанность | Эксплуатационная сложность, распределённые данные, eventual consistency |
| Модульный монолит | Проще эксплуатация и транзакции | Общий деплой/масштаб; риск размывания границ; сложнее изолировать нагрузку телефонии/AI |
| «Наносервисы»/FaaS | Тонкая гранулярность | Избыточная сложность, холодные старты, трудный аудит для госсистемы |

## Принятое решение

**Микросервисная архитектура** с **событийно-ориентированной интеграцией** через Kafka.
Дополнительно:
- **Database-per-service**, синхронное взаимодействие только через REST-контракты (**API First**).
- Внутри сервисов — **Clean Architecture** (domain/application/adapters).
- **Гексагональная** организация интеграционных адаптеров (порты/адаптеры), явно —
  в notification-service.
- **CQRS точечно** — только в ядре incident-service (команды/запросы + Redis-кэш чтения),
  без event sourcing и отдельного read-store.

## Обоснование

- **Проблема:** разные функции требуют разного масштаба и темпа изменений; монолит связывает их.
- **Альтернативы:** модульный монолит (проще, но не даёт независимого масштабирования и
  изоляции нагрузки/отказов, критичных для 112); FaaS (избыточно сложно и хуже для аудита).
- **Почему выбрано:** независимость деплоя/масштаба, чёткие границы (DDD), устойчивость.
- **Преимущества:** масштабирование ядра/AI отдельно; локализация отказов; ясные контракты.
- **Недостатки/компромиссы:** распределённые данные и eventual consistency; выше требования
  к наблюдаемости и эксплуатации. Приняты осознанно; смягчаются стандартами и платформой.
- **Почему CQRS только в ядре:** профиль чтения ядра (обстановка, статистика) отличается от
  записи; в остальных сервисах разделение избыточно (YAGNI).
- **Влияние на развитие:** новые функции — в границах контекста либо новым сервисом (по ADR/RFC).
- **Условия пересмотра:** если эксплуатационная сложность превысит выгоды на реальном профиле —
  консолидация части сервисов через RFC.

## Последствия

- Обязательны: единый периметр (Gateway), контрактное версионирование, идемпотентные консьюмеры.
- Согласованность — в конечном счёте; UI и процессы это учитывают.
- Правила зависимостей — см. Architecture.md и System Structure.
