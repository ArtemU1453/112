# ADR-012 — Распределённая трассировка

- **Статус:** Accepted
- **Дата:** 2026-07-21
- **Связи:** Observability Architecture, Architecture.md (наблюдаемость), Constitution ARTICLE 11,
  Capability `CAP-TRACE-001` (ранее Proposed → Approved)

## Контекст

Событийно-ориентированная микросервисная система (ADR-002/008) требует сквозной прослеживаемости
запросов и потоков событий между сервисами для разбора инцидентов и анализа латентности (цель Ц1).
Наблюдаемость уже включает метрики (Prometheus) и логи (Loki); недостаёт трассировки. Решение
должно быть self-hosted (air-gapped) и совместимым со стандартами.

## Рассматриваемые варианты

| Аспект | Варианты | Выбор |
|--------|----------|-------|
| Инструментирование/сбор | OpenTelemetry, проприетарные агенты | **OpenTelemetry (OTel)** |
| Backend трасс | Grafana Tempo, Jaeger, Zipkin | **Grafana Tempo** (Jaeger — совместимая альтернатива) |
| Корреляция | trace/span id в логах и заголовках | trace-context (W3C) + `correlationId` |

## Принятое решение

**OpenTelemetry** как стандарт инструментирования и сбора (реализует `CAP-TRACE-001`);
**Grafana Tempo** как backend хранения трасс (интеграция с существующей Grafana). Контекст
трассировки распространяется по W3C Trace Context (HTTP-заголовки) и заголовкам сообщений Kafka;
`traceId`/`spanId` попадают в структурированные логи (Logging Standards) для связки логов и трасс.
Сбор — через OpenTelemetry Collector внутри контура.

## Обоснование

- **Проблема:** нет сквозной трассировки в распределённой системе; сложен разбор латентности/сбоев.
- **Почему OpenTelemetry:** вендор-нейтральный стандарт (CNCF), единое инструментирование для
  Java/Python/JS, независимость от backend; переносим в изолированный контур.
- **Почему Grafana Tempo:** нативная интеграция с уже используемыми Grafana/Loki/Prometheus
  (единая панель Metrics/Logs/Traces), экономичное хранение; Jaeger оставлен совместимой
  альтернативой (OTel-совместим).
- **Почему не проприетарные агенты:** привязка к вендору, хуже для суверенного контура.
- **Недостатки/компромиссы:** накладные расходы на инструментирование и хранение трасс;
  смягчается семплированием и настройкой удержания (Observability Architecture).
- **Условия пересмотра:** смена backend трасс (Tempo↔Jaeger) не влияет на инструментирование
  (OTel) и оформляется обновлением Technology Mapping; смена стандарта инструментирования — RFC/ADR.

## Последствия

- Единая observability-архитектура Metrics/Logs/Traces/Events/Health (`docs/platform/observability-architecture.md`).
- `traceId`/`spanId` добавляются в поля логов (согласовано с Logging Standards).
- `CAP-TRACE-001` переведена в Approved (Technology Mapping); версии — в каталоге (TVMS).
