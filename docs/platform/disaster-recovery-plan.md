# Disaster Recovery Plan — ЕАСУР

- **Status:** APPROVED · Stage 3 · Подчиняется Backup Policy, Platform Operations Framework,
  Air-Gapped Standard.

## 1. Цели
- **RPO** ≤ 24 ч (оперативные БД); **RTO** ≤ 4 ч (ядро). Уточняются владельцем эксплуатации.
- Обеспечить непрерывность реагирования 112 при отказах инфраструктуры и катастрофах.

## 2. Сценарии аварий
| Сценарий | Влияние | Реакция |
|----------|---------|---------|
| Отказ узла кластера | Деградация ёмкости | Самовосстановление K8s, перепланирование подов, HPA |
| Отказ зоны/стойки | Частичная недоступность | Реплики по зонам, перенос нагрузки |
| Потеря БД/повреждение данных | Недоступность домена | Восстановление из бэкапа (Backup Policy) |
| Отказ Vault | Нет доступа к секретам | HA-Vault, процедура unseal, DR-снапшот |
| Отказ объектного хранилища | Нет бэкапов/зеркал | HA-конфигурация, репликация |
| Катастрофа площадки | Полная недоступность | Переключение на DR-площадку |
| Потеря внешней связи (air-gapped) | Изоляция | Автономная работа (AGSDS), деградационные режимы |

## 3. Процедуры восстановления
1. **Оценка** — классификация аварии (Incident Management, POF).
2. **Изоляция** — предотвращение распространения (напр., перевод в режим только чтение).
3. **Восстановление данных** — `pg_restore` из последнего валидного бэкапа; проверка целостности
   (`verify-recovery.sh`).
4. **Восстановление платформы** — повторное применение IaC/манифестов (Terraform/Kustomize/Helm)
   из VCS (`platform-bootstrap.sh`).
5. **Восстановление секретов** — из DR-снапшота Vault; unseal по регламенту.
6. **Проверка** — health/readiness, Prometheus targets, smoke-сценарии.
7. **Возврат в строй** — снятие ограничений, уведомление, RCA (POF).

## 4. Переключение на DR-площадку
- Предпосылки: реплики данных/бэкапы доступны на DR-площадке; идентичная версия платформы (EGS).
- Шаги: активация DR-кластера → восстановление данных → переключение DNS/маршрутов → проверка.
- Обратное переключение (failback) — после стабилизации основной площадки, по регламенту.

## 5. Регулярные учения
- Плановые учения по восстановлению (game day) не реже согласованной периодичности.
- Проверка восстановления бэкапов — регулярно (Backup Policy).
- Результаты фиксируются; выявленные пробелы — задачи Backlog (техдолг/риски).

## 6. Роли и коммуникации
Incident Manager координирует; Platform/Database/Security Administrators выполняют процедуры
(Platform Operations Framework). Каналы эскалации — Operational Procedures.
