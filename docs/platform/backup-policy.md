# Backup Policy — ЕАСУР

- **Status:** APPROVED · Stage 3 · Подчиняется Environment Governance Standard, DGS, Air-Gapped
  Standard.

## 1. Область
Резервируются: базы данных всех сервисов + Keycloak, конфигурация платформы (IaC/манифесты в
VCS), секреты (снапшоты Vault), объектное хранилище (артефакты/зеркала). Реальные ПДн — под
защитой и аудитом.

## 2. Цели восстановления
- **RPO** (допустимая потеря данных): ≤ 24 ч для оперативных БД (ежедневный бэкап); критичные
  БД — чаще (инкрементально/WAL) при ужесточении требований.
- **RTO** (время восстановления): ≤ 4 ч для ядра (согласовано с целью доступности Ц6).
Значения уточняются владельцем эксплуатации и фиксируются в DR Plan.

## 3. Способ
- БД: `pg_dump` (custom-формат) всех БД — `infrastructure/backup/backup.sh`; в Kubernetes —
  CronJob (`helm/emergency-112/templates/backup-cronjob.yaml`).
- Цель хранения: локальное S3-совместимое объектное хранилище (ADR-013) — без внешних облаков
  (AGSDS).
- Конфигурация/IaC: версионируется в Git (источник истины).
- Секреты: снапшоты Vault по защищённой процедуре (Secrets Management).

## 4. Периодичность и удержание
- Ежедневный полный бэкап БД (по умолчанию 02:00); удержание — 14 суток (dev/stage), 30 суток
  (prod) — параметры Helm `backup.schedule` / `backup.retentionDays`.
- Долгосрочные архивы — по регламенту хранения (EGS/правовые требования).

## 5. Проверка восстановления
- Регулярная автоматическая проверка: восстановление дампа во временную БД и контроль целостности
  — `infrastructure/scripts/verify-recovery.sh`.
- Результаты проверок журналируются; сбой — инцидент (Platform Operations Framework).

## 6. Целостность и происхождение
Каждый архив сопровождается контрольной суммой (SHA-256); для офлайн-пакетов — сведения о
происхождении и журнал изменений (AGSDS). Доступ к бэкапам — по минимальным привилегиям, с аудитом.
