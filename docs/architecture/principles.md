# Архитектурные принципы — ЕАСУР

> Приоритет SSOT: 2. Ниже Vision, выше Architecture.md.
> Статус: APPROVED · Baseline: v1.0

Для каждого принципа указано: зачем используется, где применяется, ограничения.

## SOLID
- **Зачем:** управляемость и тестируемость доменного кода; локализация изменений.
- **Где:** во всех Java-сервисах — разделение ответственности между domain/application/adapters,
  инверсия зависимостей (порты для Kafka/SMTP/HTTP).
- **Ограничения:** не доводить до избыточного дробления интерфейсов там, где нет вариативности
  (баланс с KISS/YAGNI).

## DRY
- **Зачем:** единые соглашения (обработка ошибок, ответы API, маппинг ролей) без дублирования.
- **Где:** общие подходы в Coding Standards; переиспользуемые Helm-шаблоны; единый формат
  ошибок API.
- **Ограничения:** не создавать преждевременные общие библиотеки между сервисами — это
  повышает связанность. Дублирование контрактных DTO между контекстами допустимо (автономность
  выше DRY между сервисами).

## KISS
- **Зачем:** снижать эксплуатационную сложность государственной системы.
- **Где:** stateless-сервисы, простые статусные модели, явные миграции.
- **Ограничения:** не в ущерб требуемым свойствам (безопасность, аудит, отказоустойчивость).

## YAGNI
- **Зачем:** не реализовывать неутверждённые требования (см. Constitution, ARTICLE 3).
- **Где:** объём ограничен Vision; будущие интеграции — вне текущей Baseline.
- **Ограничения:** не отменяет закладку точек расширения, предусмотренных архитектурой.

## Clean Architecture
- **Зачем:** независимость домена от фреймворков и инфраструктуры, тестируемость ядра.
- **Где:** слои внутри каждого сервиса; зависимости направлены к домену.
- **Ограничения:** для чисто интеграционных сервисов (realtime, gateway) слоистость
  минимальна — они почти не содержат домена.

## Domain-Driven Design
- **Зачем:** единый язык и корректные границы; каждый сервис = bounded context.
- **Где:** агрегаты (Incident, Unit/Assignment, Call, Station), доменные события, Ubiquitous
  Language (см. Domain Modeling Standard).
- **Ограничения:** тактические паттерны DDD применяются там, где есть богатый домен
  (incident, dispatch), а не в технических сервисах.

## Hexagonal Architecture (Ports & Adapters) — по ADR-002
- **Зачем:** заменяемость внешних каналов без изменения домена.
- **Где:** наиболее явно в notification-service (порт `ChannelSender` + адаптеры Email/SMS/Push);
  а также адаптеры Kafka/persistence во всех сервисах.
- **Ограничения:** не навязывается технически простым сервисам.

## CQRS — по ADR-002 (точечно)
- **Зачем:** разделить нагрузку чтения и записи в ядре, кэшировать чтение.
- **Где:** incident-service (`IncidentCommandService` / `IncidentQueryService` + Redis-кэш).
- **Ограничения:** без отдельного read-store и event sourcing (избыточно); применяется
  только там, где профиль чтения/записи это оправдывает.

## Event-Driven Architecture — по ADR-008
- **Зачем:** слабая связанность, устойчивость, асинхронная реакция сервисов.
- **Где:** Kafka-топики (incident.*, dispatch.*, unit.*, call.*, notification.*, audit.events).
- **Ограничения:** согласованность в конечном счёте (eventual consistency); консьюмеры
  идемпотентны и толерантны к порядку/повторам.

## API First
- **Зачем:** контракт раньше реализации; предсказуемая интеграция клиентов.
- **Где:** OpenAPI 3.1 для всех REST-эндпоинтов; версия в пути `/api/v1`.
- **Ограничения:** изменение контракта — только с версионированием (см. AGS).

## Security by Design
- **Зачем:** государственная система, ПДн, требования ИБ.
- **Где:** OIDC/JWT на периметре, RBAC-роли, аудит, минимизация привилегий, секреты вне git.
- **Ограничения:** удобство не превышает требований безопасности (Constitution, ARTICLE 10).

## Observability by Design
- **Зачем:** эксплуатируемость 24/7, быстрый разбор инцидентов.
- **Где:** health-пробы, метрики Prometheus, логи Loki, дашборды Grafana.
- **Ограничения:** отсутствие наблюдаемости делает функциональность незавершённой
  (Constitution, ARTICLE 11).

## Разрешение конфликтов принципов
При конфликте приоритет: **безопасность и корректность → сопровождаемость → простота →
производительность → лаконичность**. Спорные случаи фиксируются ADR.
