# Testing Strategy — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

Единая стратегия тестирования (Constitution ARTICLE 8; Quality Gates). Функциональность без
тестов считается незавершённой.

## Уровни тестирования (пирамида)

| Уровень | Что проверяет | Инструменты (роли; версии — в каталоге) | Где выполняется |
|---------|---------------|------------------------------------------|-----------------|
| **Unit** | Доменная логика, инварианты, чистые функции; изоляция через моки | JUnit + Mockito (Java); pytest (Python); Vitest/Jest (TS) | Локально, CI (быстро) |
| **Integration** | Взаимодействие с реальной инфраструктурой (БД, брокер) | Testcontainers (PostgreSQL/Kafka) | CI |
| **Contract** | Совместимость контрактов REST (OpenAPI) и событий Kafka между сервисами | Контрактные проверки схем/OpenAPI | CI |
| **Performance** | Латентность/пропускная способность ключевых сценариев (цели Ц1) | Нагрузочные сценарии | Performance-окружение |
| **Security** | Уязвимости, доступ (RBAC), секреты | SAST/DAST, dependency/container scan | CI, Security-окружение |
| **E2E** | Сквозные сценарии (вызов→карточка→наряд→реальное время) | Автотесты поверх собранной системы | Staging/QA |

## Принципы
- **Изоляция:** Unit-тесты не зависят от сети/инфраструктуры; Integration используют
  Testcontainers, а не общие внешние сервисы.
- **Детерминизм:** тесты повторяемы; без зависимости от времени/порядка/внешнего состояния.
- **Доменный фокус:** приоритет — покрытие доменных инвариантов (статусные модели, автодиспетчеризация,
  валидация переходов) и критических путей.
- **Идемпотентность консьюмеров** проверяется интеграционными тестами (повторная доставка события).

## Требования к покрытию
- Обязателен Unit-уровень для доменного/прикладного кода; Integration — для персистентности и
  Kafka-обмена; Contract — при наличии межсервисных контрактов.
- Порог покрытия доменного кода — согласно Quality Gates; снижение ниже порога блокирует слияние.
- Отсутствие теста для функциональности допускается только с явным обоснованием требованиями
  (Requirements Management Standard).

## Данные для тестов
- Только синтетические/обезличенные данные (Environment Governance Standard); реальные ПДн в
  тестах запрещены.

## Организация
- Тесты сервиса — рядом с кодом сервиса (`src/test/...`, `tests/` для Python).
- Кросс-сервисные Contract/E2E — в каталоге `tests/` монорепозитория.
- Отчёты (JUnit XML) публикуются в CI (build/test workflow).

## Definition of Done (тестовая часть)
Тесты требуемых уровней написаны и проходят; отчёты сформированы; Quality Gates по тестам зелёные.
