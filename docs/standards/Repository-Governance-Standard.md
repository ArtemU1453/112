# Repository Governance Standard (RGS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

## Цель
Определить структуру репозитория, владение кодом, управление ветками и файлами, а также
политику репозитория. Обеспечить единообразие и управляемость монорепозитория.

## Область применения
Весь монорепозиторий ЕАСУР: сервисы, приложения, библиотеки, инфраструктура, документация.

## Определения
- **Монорепозиторий** — единый репозиторий, содержащий все компоненты системы.
- **Модуль** — сервис, приложение или библиотека с назначенным владельцем.

## Структура репозитория

### Допустимые каталоги верхнего уровня
| Каталог | Назначение |
|---------|-----------|
| `apps/` | Клиентские приложения (АРМ, мобильные) |
| `services/` | Backend-микросервисы (bounded contexts) |
| `libraries/` | Общие библиотеки/сборочные артефакты (parent, BOM) |
| `docs/` | Архитектурная и инженерная документация (SSOT) |
| `scripts/` | Скрипты автоматизации разработки/проверок |
| `docker/` | Compose-файлы окружений разработки/тестов/инструментов/мониторинга |
| `helm/` | Helm-чарты развёртывания |
| `kubernetes/` | Kustomize base/overlays по окружениям |
| `tools/` | Вспомогательные инструменты и конфигурации |
| `config/` | Централизованная конфигурация (каталог версий, quality) |
| `environments/` | Шаблоны конфигураций окружений (без секретов) |
| `tests/` | Кросс-сервисные тесты (contract/e2e) |
| `infrastructure/` | Инициализация БД, IdP realm, мониторинг, backup |
| `.github/` | Конфигурация GitHub (шаблоны, workflows, метаданные) |
| `.devcontainer/` | Воспроизводимая среда разработки |
| `.vscode/` | Общие настройки редактора (committed subset) |

### Правила создания/удаления/архивирования
- Новый каталог верхнего уровня — только через RFC + ADR (Architecture Freeze).
- Удаление каталога модуля — через задачу Backlog с анализом влияния; история сохраняется в VCS.
- Архивирование модуля — перенос в статус `archived` (документируется), без удаления истории.

## Владение кодом
- Владение определяется файлом `CODEOWNERS` (в корне/`.github/`).
- У каждого модуля — назначенный владелец и обязательные ревьюеры.
- Смена владельца — через PR с одобрением текущего владельца и ответственного за архитектуру.
- Минимальное число одобрений PR — **1** (для архитектурно значимых изменений — дополнительно
  архитектурный ревью). Ветки `main`/`develop` защищены (см. «Управление ветками»).

## Управление ветками
- Жизненный цикл и именование — по Git Strategy (`feature/*`, `bugfix/*`, `hotfix/*`,
  `release/*`, `develop`, `main`).
- Защита основных веток: запрет прямого push, обязательный PR, зелёный CI, ревью.
- Слияние — squash/merge по политике; ветка удаляется после слияния.

## Управление файлами
- Допустимые форматы — исходный код, конфигурации, документация (текстовые).
- Ограничение размера файла в git — крупные бинарники не хранятся напрямую.
- Бинарные/крупные артефакты — через Git LFS при необходимости (по согласованию).
- Правила текстовых атрибутов и окончаний строк — `.gitattributes`, `.editorconfig`.

## Политика репозитория
- **Запрет хранения секретов** (Git/Dockerfile/образы/конфиги) — только плейсхолдеры/шаблоны
  (Environment Governance Standard). Проверяется секрет-сканером в CI.
- **Запрет хранения сгенерированных артефактов** (build output, `node_modules`, дампы) — `.gitignore`.
- Конфигурации — централизованно и версионируемо (`config/`, `environments/`).
- Устаревшие компоненты удаляются/архивируются через управляемый процесс.

## Рекомендуемые практики
- Атомарные коммиты (Conventional Commits); один PR — одна задача.
- Согласованность структуры проверяется автоматически (architecture validation).

## Исключения
Отклонения от структуры — только с обоснованием и ссылкой на ADR.

## Процесс изменения документа
Через RFC + ADR при влиянии на структуру/правила; иначе — PR с ревью и записью в журнал версий.
