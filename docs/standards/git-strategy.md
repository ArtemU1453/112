# Git Strategy — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

Модель ветвления, правила Pull Request и Code Review.

## Ветви

| Ветвь | Назначение | Источник | Слияние в |
|-------|------------|----------|-----------|
| `main` | Стабильный релизный код (production) | — | — |
| `develop` | Интеграционная ветвь разработки | `main` | `main` (через release) |
| `feature/*` | Новая функциональность (одна задача) | `develop` | `develop` |
| `bugfix/*` | Исправление дефекта (не срочное) | `develop` | `develop` |
| `release/*` | Стабилизация релиза | `develop` | `main` и `develop` |
| `hotfix/*` | Срочное исправление в проде | `main` | `main` и `develop` |

Именование: `feature/INC-001-incident-aggregate`, `bugfix/AUTH-125`, `hotfix/DISP-007`,
`release/v1.0.0`. Одна задача — одна ветка (Engineering Handbook, Section 4).

> Примечание: в средах автоматизированной разработки допускается работа в специально
> назначенной ветке задания; правила PR/ревью ниже применяются к ней так же.

## Коммиты

Conventional Commits: `<type>(<scope>): <description>`. Типы — feat, fix, refactor, perf, docs,
test, build, ci, style, chore, revert. Коммиты атомарные; сообщение на английском в заголовке,
пояснение — при необходимости.

## Pull Request

Требования к PR перед слиянием:
- Ссылка на задачу Backlog; описание изменения и влияния.
- Зелёный CI: сборка, тесты (unit/IT), статический анализ, security scan.
- Пройден Code Review (минимум один ревьюер; для архитектурно значимых — архитектурный ревью).
- Обновлена документация (Constitution, ARTICLE 9), при изменении архитектуры — RFC/ADR.
- Нет снижения покрытия ниже порога Quality Gates; нет новых критических предупреждений анализа.
- Обратная совместимость соблюдена или оформлена через RFC (ARTICLE 6).

Слияние — squash или merge-commit по политике репозитория; ветка удаляется после слияния.

## Code Review (чек-лист)

- Соответствие ADR/RFC и Coding Standards.
- Архитектура: границы контекста, правила зависимостей, отсутствие обращения к чужим БД.
- Корректность и обработка ошибок; безопасность (нет секретов/ПДн в коде и логах).
- Тесты присутствуют и осмысленны; производительность приемлема.
- Читаемость, отсутствие дублирования и мёртвого кода.

## Релизы

- Семантическое версионирование `MAJOR.MINOR.PATCH`.
- Тег релиза на `main`; образы тегируются SHA и каналом (`stable`).
- Changelog ведётся для релизов и изменений Baseline.
