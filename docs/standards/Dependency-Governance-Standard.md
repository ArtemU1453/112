# Dependency Governance Standard (DepGS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0
- Примечание: аббревиатура «DGS» в проекте закреплена за Database Governance Standard; данный
  документ обозначается **DepGS** во избежание коллизии.

## Цель
Определить политику управления внешними и внутренними зависимостями: единый источник версий,
лицензирование, безопасность, обновления.

## Область применения
Все зависимости всех модулей (Java/Maven, Node/npm, Python/pip) и внутренние библиотеки.

## Определения
- **Прямая зависимость** — объявленная модулем.
- **Транзитивная зависимость** — привнесённая прямой.
- **Внутренняя библиотека** — артефакт из `libraries/`.

## Политика зависимостей
- **Единый источник версий:** версии инструментов/платформ — в каталоге версий
  (`config/toolchain/versions.yaml`); версии библиотек управляются централизованно через
  **Dependency Management / BOM** (`libraries/build/bom`) и parent-конфигурацию
  (`libraries/build/parent`).
- **Централизованное управление:** запрещены локальные (пер-модульные) переопределения версий
  библиотек, управляемых BOM, без обоснования.
- **Правила обновления** — см. «Политику обновлений».

## Проверка лицензий
- Допустимые лицензии по умолчанию: **Apache-2.0**, **MIT**, **BSD** (2/3-clause).
- Прочие лицензии (LGPL, MPL, EPL и т. п.) — рассмотрение в индивидуальном порядке с оценкой
  совместимости и рисков; copyleft-строгие (GPL/AGPL) для распространяемых компонентов —
  запрещены без отдельного решения.
- Проверка лицензий автоматизирована в CI (license checking) и фиксируется в SBOM.

## Проверка безопасности
Обязательные проверки (CI, `.github/workflows/dependency-scan.yml`, `security.yml`):
- известные **CVE** в прямых и транзитивных зависимостях;
- **критические/высокие** уязвимости блокируют слияние;
- устаревшие библиотеки (флагирование);
- генерация **SBOM** (CycloneDX/SPDX) для прослеживаемости.

## Политика обновлений
- Периодические обновления (регулярный цикл) + внеплановые при критических CVE.
- После обновления — прогон тестов требуемых уровней и Quality Gates.
- Откат обновления при регрессе; фиксация в журнале и задаче Backlog.

## Управление внутренними библиотеками
- Единое версионирование (SemVer) артефактов `libraries/`.
- Правила публикации — в репозиторий артефактов (Release Governance Standard).
- Совместимость: несовместимые изменения — мажорная версия; потребители мигрируют по плану.

## Рекомендуемые практики
- Минимизировать число прямых зависимостей; предпочитать зрелые сопровождаемые библиотеки.
- Избегать дублирующих библиотек одной возможности.

## Исключения
Нестандартная/непроверенная зависимость — только с обоснованием, анализом альтернатив, оценкой
рисков и ссылкой на ADR.

## Процесс изменения документа
RFC + ADR при влиянии на политику; иначе — PR с ревью и записью в журнал версий.

## Связанные стандарты
Toolchain Governance Standard · Toolchain Version Management Standard · Release Governance
Standard · Repository Governance Standard · Coding Standards · Quality Gates.
