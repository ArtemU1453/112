# Deprecation Policy Standard — ЕАСУР

- **Status:** APPROVED · Stage 11 · Согласован с ADR-025, Component Lifecycle Standard, API Versioning
  Standard, Upgrade Compatibility Standard, Knowledge Governance Standard.

## Назначение
Правила объявления устаревания и вывода компонентов/API/технологий (11.1, 11.4). Прекращение поддержки
без утверждённого плана миграции запрещено (запрет этапа).

## Принципы
- **Предсказуемость:** депрекация объявляется заранее с указанием сроков и версии вывода.
- **План миграции обязателен:** до вывода публикуется путь перехода (альтернатива, инструменты, сроки).
- **Коммуникация:** уведомление потребителей (внутренних/внешних через реестр интеграций), фиксация в
  release notes и базе знаний.
- **Неудаление знаний:** документация выведенного помечается статусом (Deprecated/Retired), не удаляется.

## Жизненный цикл депрекации
```
Active → Deprecated (объявление + сроки + план миграции) → Sunset (только безопасность/критфиксы) → Retired
```
- Минимальные окна поддержки задаются по критичности и типу (API/библиотека/технология/сервис).
- Досрочный вывод — только при критической угрозе безопасности, с ускоренной миграцией и решением ARB.

## Процесс
Предложение о депрекации → оценка влияния (потребители/интеграции) → решение ARB (RFC/ADR при
архитектуре) → объявление и сроки → миграция потребителей → вывод (Retired) → фиксация.

## Definition of Done
Определены принципы, жизненный цикл и процесс депрекации с обязательным планом миграции.
