# Environment Governance Standard (EGS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

## Цель
Определить единые правила проектирования, конфигурации, эксплуатации и сопровождения всех
окружений. Окружения максимально воспроизводимы и различаются только действительно необходимыми
параметрами.

## Область применения
Все окружения проекта — от локальной разработки до production и DR.

## Классификация окружений

| Окружение | Назначение | Владелец | Реальные ПДн |
|-----------|-----------|----------|--------------|
| Local Development | Разработка на машине инженера (Dev Container) | Разработчик | Запрещены |
| Development | Общая интеграция ветки разработки | Команда разработки | Запрещены |
| Integration | Проверка межсервисной интеграции | Команда разработки | Запрещены |
| QA | Функциональное тестирование | QA | Запрещены |
| Performance Testing | Нагрузочное тестирование | QA/Perf | Запрещены (синтетика) |
| Security Testing | Тестирование безопасности | Security | Запрещены |
| UAT | Приёмочное тестирование заказчиком | Владелец продукта | По отдельной политике |
| Staging | Предпрод, идентичен prod | Эксплуатация | Обезличенные/по политике |
| Production | Промышленная эксплуатация | Эксплуатация | Да (с защитой) |
| Disaster Recovery | Резервная площадка | Эксплуатация | Да (реплика prod) |
| Sandbox (опц.) | Эксперименты | По назначению | Запрещены |

Для каждого окружения описываются: назначение, владелец, состав компонентов, ограничения,
допустимые операции.

## Конфигурация
Для каждого окружения определяются: переменные окружения; секреты (ссылки, не значения);
параметры БД/Kafka/Redis; параметры логирования/мониторинга; feature flags; ограничения доступа.
Конфигурация хранится централизованно и версионируемо (`environments/`, `config/`); значения по
умолчанию и шаблоны — без секретов.

## Управление секретами
Запрещено хранить секреты в Git, Dockerfile, образах, открытых конфигурациях. Механизм хранения —
внешний секрет-менеджер (возможность `CAP-SECRETS-001`, Vault/эквивалент; утверждается ADR).
Правила ротации секретов определяются политикой ИБ; в Kubernetes — через Secret/External Secrets.

## Продвижение изменений
```
Local → Development → Integration → QA → UAT → Staging → Production
```
Пропуск стадий запрещён без отдельного утверждения (Governance Checkpoint).

## Синхронизация окружений
- **Идентичны во всех окружениях:** версии инструментов (каталог), схемы БД (миграции),
  контракты API/событий, структура конфигурации.
- **Могут отличаться:** масштаб (реплики/ресурсы), endpoint'ы инфраструктуры, feature flags,
  уровни логирования, лимиты.
- Согласованность проверяется автоматически перед развёртыванием (см. «Проверка соответствия»).

## Данные
Тестовые/обезличенные/синтетические данные вне Production. Использование реальных ПДн вне
Production запрещено, кроме случаев, разрешённых отдельной политикой (UAT/DR).

## Наблюдаемость
Для каждого окружения: уровень логирования, сбор метрик, трассировка, health checks, alerting
(детали — Logging Standards, Architecture.md). Production/Staging — полный набор.

## Проверка соответствия (перед развёртыванием)
Автоматически: корректность конфигурации; наличие обязательных переменных; отсутствие
запрещённых секретов; соответствие утверждённым версиям (каталог); успешное прохождение CI/CD.

## Резервное копирование
Где применимо (Staging/Production/DR): периодичность, срок хранения, процедура восстановления,
регулярная проверка восстановления (см. `infrastructure/backup/`, Helm CronJob, operations.md).

## Управление изменениями
Изменение конфигурации окружения: фиксируется в VCS; проходит review; имеет журнал изменений;
обратимо (rollback).
