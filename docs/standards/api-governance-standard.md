# API Governance Standard (AGS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0 · Приоритет SSOT: 7 (API Contracts)

Единые правила проектирования REST API. Все API выглядят и ведут себя единообразно (API First,
Constitution ARTICLE 12).

## Стиль и контракты

- Архитектурный стиль — REST поверх HTTP; ресурсы — существительные во множественном числе,
  `kebab-case`.
- Контракт — **OpenAPI 3.1** для каждого сервиса; контракт первичен по отношению к реализации.
- Базовый путь через Gateway: `/api/v1/<resource>`.

## Версионирование

- Версия в пути: `/api/v1`, `/api/v2`. Обратно несовместимое изменение — новая мажорная версия.
- Аддитивные изменения (новые необязательные поля/эндпоинты) — без смены версии.
- Старая версия поддерживается до согласованного вывода из эксплуатации (deprecation policy).

## HTTP-коды (базовый набор)

| Код | Применение |
|-----|-----------|
| 200 | Успешный GET/обновление с телом |
| 201 | Создание ресурса (+ `Location`) |
| 204 | Успех без тела |
| 400 | Ошибка валидации запроса |
| 401 | Не аутентифицирован |
| 403 | Нет прав (RBAC) |
| 404 | Ресурс не найден |
| 409 | Конфликт (нарушение инварианта, оптимистическая блокировка, недопустимый переход статуса) |
| 422 | Семантически некорректный запрос |
| 429 | Превышение лимита частоты |
| 5xx | Внутренние ошибки |

## Обработка ошибок

- Единый формат — **RFC 7807** (`application/problem+json`): `type`, `title`, `status`,
  `detail`, `instance`; для валидации — список нарушений по полям.
- Без утечки внутренних деталей/стека и ПДн в теле ошибки.

## Пагинация, фильтрация, сортировка

- Пагинация: `?page=<0..>&size=<n>`; ответ содержит метаданные (total, page, size).
- Фильтрация: явные query-параметры по полям; сложные выборки — через спецификации.
- Сортировка: `?sort=<field>,<asc|desc>` (допускается несколько).

## Идемпотентность и безопасность методов

- GET/PUT/DELETE — идемпотентны; POST создаёт ресурс. Для критичных POST — поддержка ключа
  идемпотентности при необходимости (по требованию сценария).
- Все изменяющие операции проходят аутентификацию (JWT) и авторизацию (RBAC-роли).

## Совместимость

- Изменение контракта проверяется на обратную совместимость (Compatibility Check, RFC).
- Несовместимое изменение требует новой версии API и обновления клиентов и документации.

## Документирование

- Каждый эндпоинт — с описанием, параметрами, кодами ответов и примерами в OpenAPI.
- OpenAPI поддерживается синхронно с реализацией (Document Synchronization).
