# Requirements Management Standard (RMS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

Определяет жизненный цикл требований, идентификаторы, трассируемость, управление изменениями
и правила декомпозиции.

## Жизненный цикл требования

```
Proposed → Analyzed → Approved → Implemented → Verified → Closed
                    ↘ Rejected / Deferred
```

Требование фиксируется до реализации; изменение утверждённого требования проходит управление
изменениями (см. ниже) и, при влиянии на архитектуру, — RFC.

## Классификация

- **Функциональные (FR):** поведение системы (приём вызова, создание карточки, автодиспетчеризация).
- **Нефункциональные (NFR):** доступность, производительность, безопасность, наблюдаемость,
  масштабируемость, удобство эксплуатации (связаны с целями Vision Ц1–Ц6).
- **Ограничения:** платформенные/правовые/языковые (см. Vision §6).

## Идентификаторы

`REQ-<CTX>-<NNN>` — например `REQ-INC-001`, `REQ-NFR-003`. Идентификатор неизменяем; при
существенном изменении смысла заводится новое требование со ссылкой на предыдущее.

## Трассируемость (обязательна)

```
Требование (REQ) → Feature → Story → Task → Код → Тест (Unit/IT/E2E) → Документация
```

Каждый элемент реализации ссылается вверх на требование; каждое требование имеет хотя бы один
проверяющий тест или явное обоснование его отсутствия (DoD, ARTICLE 8). Матрица трассируемости
поддерживается в Backlog.

## Управление изменениями требований

1. Заявка на изменение с причиной и влиянием.
2. Анализ влияния на связанные требования, контракты, данные, безопасность.
3. Решение (Approved/Rejected/Deferred).
4. При влиянии на архитектуру — RFC + ADR.
5. Обновление трассируемости и документации (Document Synchronization).

## Правила декомпозиции

- Требование декомпозируется до Story/Task, каждая из которых удовлетворяет INVEST
  (независима, обсуждаема, ценна, оцениваема, мала, тестируема).
- Критерии приёмки формулируются проверяемо (предпочтительно Given/When/Then).
- Нефункциональные требования выражаются измеримыми показателями (пороги, SLA), а не общими
  формулировками.
- Отсутствующая информация не домысливается: пробелы фиксируются как Spike/вопрос (Vision §8).
