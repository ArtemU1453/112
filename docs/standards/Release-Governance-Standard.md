# Release Governance Standard (RLS) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

## Цель
Определить жизненный цикл релиза, версионирование, политику релизов, changelog, подписание и
хранение артефактов, политику отката.

## Область применения
Все выпускаемые артефакты: образы сервисов, Helm-чарты, пакеты внутренних библиотек.

## Определения
- **Релиз** — согласованный набор артефактов с единой версией.
- **Артефакт** — собранный образ/пакет/чарт, публикуемый в репозиторий артефактов.

## Жизненный цикл релиза
```
Development → Release Candidate → Testing → Approved → Production → Maintenance → Archived
```
Переходы последовательны; продвижение в Production — ручной gate (ADR-006/ADR-009) после
прохождения Quality Gates и проверок окружений.

## Версионирование
- **Semantic Versioning**: `MAJOR.MINOR.PATCH`.
  - MAJOR — несовместимые изменения (контракты API/событий/данных) — требуют RFC.
  - MINOR — обратно совместимая функциональность.
  - PATCH — обратно совместимые исправления.
- Теги релизов на `main` (см. Tag Strategy в Git Strategy). Конкретные версии инструментов —
  из каталога версий (TVMS), не в теле стандарта.

## Политика релизов
- **Плановые релизы** — по циклу разработки.
- **Внеплановые релизы** — по необходимости.
- **Hotfix** — срочное исправление в Production (ветка `hotfix/*`), с обратным слиянием в `develop`.
- **Emergency release** — критический инцидент/уязвимость; ускоренный, но с обязательным
  changelog, отслеживаемостью и последующим ретроспективным ревью.

## Changelog
Обязательная структура на каждый релиз:
- новые возможности; исправления; изменения безопасности; изменения зависимостей;
  известные ограничения.
Формат — «Keep a Changelog»-совместимый; ведётся в репозитории и в описании релиза.

## Подписание артефактов
- Правила подписи образов/пакетов (подпись артефактов релиза).
- Проверка целостности при доставке; хранение контрольных сумм рядом с артефактами.

## Хранение артефактов
- Репозиторий артефактов (реестр образов/пакетов).
- Срок хранения и правила удаления/архивирования — по политике эксплуатации; релизные версии
  сохраняются дольше промежуточных.

## Политика отката
Каждый релиз обязан иметь: критерии отката; последовательность отката; проверку после отката;
документирование результата (согласовано с Environment Governance Standard и Migration Plan).

## Рекомендуемые практики
- Неизменяемые артефакты (immutable), тег = SHA + канал.
- Автоматизация выпуска через `.github/workflows/release.yml`.

## Исключения
Отклонения от процесса релиза — только с обоснованием и ссылкой на ADR.

## Процесс изменения документа
RFC + ADR при влиянии на процесс релиза; иначе — PR с ревью и записью в журнал версий.
