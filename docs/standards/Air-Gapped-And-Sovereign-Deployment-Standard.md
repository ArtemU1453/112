# Air-Gapped & Sovereign Deployment Standard (AGSDS) — ЕАСУР

- **Status:** APPROVED · Stage 3 · Согласован с Project Constitution, Architecture Baseline, EAGF.

## Цель
Определить требования к развёртыванию, эксплуатации и сопровождению системы в полностью
изолированных (air-gapped) и суверенных инфраструктурах. Система должна работать без постоянного
подключения к сети Интернет.

## Область применения
Локальные ЦОД; государственные вычислительные центры; резервные площадки; мобильные командные
пункты (если применимо); тестовые/учебные полигоны с ограниченной связностью.

## Основные требования
Платформа должна поддерживать: установку без доступа к Интернету; обновление из локальных
репозиториев; локальное хранение контейнерных образов, пакетов и документации; автономную работу
мониторинга, журналирования и системы управления конфигурацией.

## Локальные репозитории (зеркала)
Внутри контура доступны зеркала: контейнерные образы (локальный OCI-реестр, `CAP-REGISTRY-001`),
зависимости сборки, библиотеки, внутренние пакеты, эксплуатационная документация. Все критически
важные зависимости доступны из внутренней инфраструктуры. Подготовка узлов/реестра — Ansible
(`infrastructure/ansible/roles/{container-runtime,local-registry}`).

## Обновление (офлайн)
Процедура без внешних сетей: подготовка пакета обновления (`build-offline-bundle.sh`) → проверка
целостности (SHA-256) → проверка подлинности (подпись, если применяется) → импорт
(`offline-update.yml`) → журналирование процесса → возможность отката. Стадии зафиксированы в
Operational Procedures.

## Целостность и происхождение
Каждый поставляемый артефакт сопровождается: контрольной суммой; цифровой подписью (если принята
архитектурой); сведениями о происхождении (provenance); журналом изменений (CHANGELOG). Проверка —
перед импортом.

## Устойчивость и деградационные режимы
При отсутствии внешней связи система сохраняет работоспособность в рамках утверждённых требований.
Определяются: допустимые ограничения; деградационные режимы (напр., отложенная синхронизация
внешних данных); порядок восстановления после появления связи. Внешний egress в air-gapped
запрещён (Network Architecture).

## Мониторинг и журналирование
Работают автономно (Observability Architecture). Передача данных во внешние системы не является
условием функционирования.

## Управление временем
Синхронизация через локальные NTP-источники (`chrony`, роль `common`). Допустимые источники —
внутренние NTP; при недоступности — удержание по drift-файлу и оповещение (Infrastructure
Architecture).

## Резервное копирование
Выполняется без внешних облаков: локальное S3-совместимое хранилище (ADR-013). Периодичность,
проверка восстановления и хранение архивов — Backup Policy.

## Документация
Все эксплуатационные документы/инструкции доступны локально (в репозитории/офлайн-пакете). Работа
персонала не зависит от внешних веб-ресурсов.

## Проверка соответствия (перед вводом)
- ✓ установка без доступа к Интернету;
- ✓ все зависимости доступны локально;
- ✓ обновление из локальных источников;
- ✓ мониторинг/журналирование автономны;
- ✓ резервное копирование локально;
- ✓ документация полностью доступна внутри инфраструктуры.

## Definition of Done
- ✓ определены требования к изолированному развёртыванию;
- ✓ описаны процедуры автономной установки и обновления;
- ✓ определены требования к локальным репозиториям;
- ✓ определены требования к автономной эксплуатации;
- ✓ документ согласован с Constitution, Architecture Baseline и EAGF.
