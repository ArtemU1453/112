# Technology Abstraction Policy (TAP) — ЕАСУР

- **Status:** APPROVED · Baseline: v1.0

## Цель

Обеспечить независимость архитектурной и инженерной документации от конкретных программных
продуктов и их версий. Стандарты описывают требования к **возможностям (capabilities)**; выбор
конкретной технологии фиксируется в **ADR**; конкретные версии — в каталоге версий
(`config/toolchain/versions.yaml`, TVMS).

## Основной принцип

- Архитектурные/инженерные документы отвечают на вопрос: **«Какая возможность необходима?»**
- ADR отвечает на вопрос: **«Какой продукт реализует эту возможность?»**
- Каталог версий отвечает: **«Какая версия продукта утверждена?»**

## Capability First

Все требования формулируются через возможности, а не названия продуктов. Запрещается
использовать название конкретного продукта в качестве требования. Перечень возможностей и их
привязка к ADR/технологиям — в `docs/architecture/technology-mapping.md`.

## Правило ссылок

В документах вместо названий технологий используются ссылки на **Capability ID** и/или **ADR**.
Пример: «Использовать утверждённый брокер сообщений (см. `CAP-MSG-001` и ADR-008)».

Исключение: сами **ADR** называют конкретный продукт (это их назначение), но **не фиксируют
номер версии** — версия берётся из каталога (TVMS ARCHITECTURE RULE).

## Изменение технологии

Замена технологии допускается только после: (1) анализа альтернатив; (2) оценки совместимости;
(3) оценки стоимости миграции; (4) подготовки RFC; (5) утверждения нового ADR; (6) обновления
Technology Mapping. После этого прочие документы продолжают ссылаться на Capability ID и не
требуют изменений, если сами требования не изменились.

## Проверка (Documentation Review / CI)

- Отсутствие необоснованных упоминаний конкретных продуктов в capability-документах.
- Наличие ADR для каждой утверждённой технологии.
- Корректность ссылок на Capability ID.
- Отсутствие захардкоженных номеров версий вне каталога.

Автоматизированная часть — `.github/workflows/architecture-validation.yml` и
`.github/workflows/documentation-validation.yml`.

## Definition of Done

- ✓ Создан Technology Abstraction Policy.
- ✓ Создан Technology Mapping (`docs/architecture/technology-mapping.md`).
- ✓ Инженерные стандарты ссылаются на Capability ID/ADR вместо жёстко заданных технологий.
- ✓ Процесс выбора и замены технологий описан и согласован (RFC + ADR + Technology Mapping).
