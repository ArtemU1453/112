# Incident Lifecycle Standard — ЕАСУР

- **Status:** APPROVED · Stage 5 · Согласован с Emergency Domain Model, Operational State Machine,
  DMS. Синхронизирован с incident-service.

## Жизненный цикл карточки (5.5)
Статусы — `IncidentStatus` (Operational State Machine §1): RECEIVED → CLASSIFIED → DISPATCHED →
IN_PROGRESS → RESOLVED → CLOSED (+ CANCELLED; RESOLVED→IN_PROGRESS для возврата в работу).

## Состав карточки
Уникальный номер; тип (`IncidentType`) и приоритет (`IncidentPriority`); адрес/координаты;
описание; число пострадавших; **версии** (оптимистическая блокировка); **журнал изменений**
(история); **вложения**; **связанные вызовы** и **события**; **участники**; **временная шкала**;
**история обработки**.

## Правила изменения (обязательные)
- **Любое изменение карточки фиксируется** (версия + запись истории + Operational Audit) —
  изменения в обход журнала **запрещены**.
- Недопустимые переходы статусов отклоняются (HTTP 409); конкурентные правки — оптимистическая
  блокировка (HTTP 409).
- Классификация/приоритизация — Incident Classification Standard; предложения ИИ требуют
  подтверждения (AI Assistance Policy).

## Создание и источники
Карточка создаётся автоматически по `call.analyzed` (из подтверждаемых данных) либо вручную
диспетчером. Публикуются `incident.created`/`incident.updated` (ADR-008) → диспетчеризация,
реальное время, аудит, уведомления.

## Трассируемость
Полная связь вызов→сущности→карточка→наряд→завершение (Operational Timeline 5.9; Emergency
Operations Platform Architecture §4).

## Definition of Done
Определены статусы/версии/история/вложения/связи/участники/временная шкала; фиксация всех изменений.
