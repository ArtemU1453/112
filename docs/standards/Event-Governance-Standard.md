# Event Governance Standard — ЕАСУР

- **Status:** APPROVED · Stage 4 · ADR-008 · Согласован с AGS, Messaging Architecture.

## Цель
Единые правила проектирования, именования, версионирования и эволюции событий и команд шины
(Messaging Platform).

## Именование
- Топики/события: `<context>.<event>` в прошедшем времени для событий (`incident.created`),
  повелительно для команд. Несовместимая эволюция — новый топик с версией (`<...>.v2`).
- Ключ партиции задаёт порядок; выбирается по агрегату.

## Схема и совместимость
- Полезная нагрузка эволюционирует **аддитивно**; потребители толерантны к неизвестным полям.
- Несовместимые изменения — новая версия события; переходный период с двойной публикацией при
  необходимости.

## Надёжность
- Идемпотентность потребителей (`IdempotencyStore`); Retry Policy; Dead Letter Queue.
- Гарантия at-least-once; эффективно-once через ключ идемпотентности.

## Каталог событий
Каждое событие документируется: назначение, продюсер, потребители, схема, версия (Messaging
Architecture, System Structure). Новое событие/поток — с обновлением каталога.

## Управление
Изменения событий — через PR/ревью; межсервисные контракты событий — как публичные (обратная
совместимость). Мониторинг lag/DLQ (Operational Telemetry).

## Definition of Done
Определены именование, схема/совместимость, гарантии доставки и каталог событий.
