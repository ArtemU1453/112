# Vision — ЕАСУР

> Единая автоматизированная система управления реагированием экстренных служб
> Республики Беларусь (система обслуживания вызовов 112/МЧС).
>
> Статус: APPROVED · Baseline: v1.0 · Приоритет SSOT: 1 (высший)

## 1. Назначение системы

ЕАСУР — государственная информационная система приёма и обработки обращений граждан
о происшествиях, автоматизированной классификации вызовов, формирования карточек
происшествий, диспетчеризации сил и средств экстренных служб и контроля реагирования
в реальном времени.

Система обслуживает единый номер вызова экстренных служб **112** и координирует
взаимодействие подразделений МЧС, скорой медицинской помощи, милиции, аварийных и
коммунальных служб.

## 2. Цели

| # | Цель | Измеримый критерий |
|---|------|--------------------|
| Ц1 | Сократить время от приёма вызова до назначения наряда | Медиана назначения наряда ≤ 60 с для приоритетов CRITICAL/HIGH |
| Ц2 | Исключить потерю обращений | 0 необработанных вызовов; полный аудит жизненного цикла обращения |
| Ц3 | Обеспечить единую оперативную картину | Обновление обстановки на АРМ диспетчера ≤ 2 с (WebSocket) |
| Ц4 | Автоматизировать первичную обработку речи | Классификация и извлечение адреса/пострадавших из транскрипта вызова |
| Ц5 | Обеспечить прослеживаемость и отчётность | Неизменяемый журнал аудита всех действий; аналитические выборки |
| Ц6 | Обеспечить доступность 24/7 | Целевая доступность ядра ≥ 99.9%, горизонтальное масштабирование |

## 3. Границы проекта (Scope)

### В границах
- Приём и учёт вызовов (телефония), запись и передача на транскрибацию.
- Автоматическая классификация обращений, извлечение сущностей (адрес, телефон,
  число пострадавших), рекомендации по нарядам (AI).
- Карточка происшествия: создание, статусная модель, история, поиск.
- Диспетчеризация: учёт подразделений, автоподбор ближайшего наряда, назначения.
- Геоинформационные функции: геокодирование, ближайшие станции, зоны ответственности.
- Уведомления (Email/SMS/Push) экипажам и службам.
- Единая точка входа, аутентификация/авторизация (OIDC + RBAC), аудит.
- Реальное время (WebSocket), наблюдаемость, эксплуатационные процедуры.
- Развёртывание: Docker Compose (dev/stage) и Kubernetes/Helm (prod).

### Вне границ (на текущей Baseline)
- Юридически значимый документооборот и интеграция с внешними государственными
  реестрами (выносится в отдельный этап «Интеграции», требует отдельных ADR).
- Мобильные приложения экипажа/гражданина (заявлены как будущие приложения, не входят
  в MVP диспетчера).
- Биллинг, кадровый учёт, бухгалтерия.

| Роль | Интерес |

## 5. Основные сценарии использования

1. **Приём вызова → карточка → наряд.** Входящий вызов регистрируется телефонией,
   запись передаётся AI-сервису; по результатам анализа автоматически формируется
   карточка происшествия; для приоритетов CRITICAL/HIGH при наличии координат
   автоматически подбирается ближайший наряд; экипаж получает уведомление; обстановка
   обновляется в реальном времени.
2. **Ручное создание карточки** диспетчером с классификацией, адресом, приоритетом.
3. **Диспетчеризация вручную**: выбор подразделения, назначение, смена статусов.
4. **Мониторинг обстановки**: карта происшествий и подразделений, таблицы, статистика.
5. **Аудит и разбор**: поиск по журналу событий, восстановление хронологии инцидента.
6. **Администрирование**: заведение пользователей, назначение ролей, блокировка.

## 6. Ограничения

- Государственная система: повышенные требования к безопасности, аудиту, доступности.
- Русскоязычная предметная область (адреса, речь, интерфейс).
- Геоданные Республики Беларусь; расчёты расстояний и зон ответственности.
- Развёртывание в контролируемом контуре (on-premise Kubernetes) как целевая среда.
- Обработка персональных данных заявителей — минимизация, защита, аудит доступа.

## 7. Критерии успеха

- Достигнуты измеримые цели Ц1–Ц6.
- Реализованы основные сценарии 1–6 с end-to-end прослеживаемостью.
- Система разворачивается воспроизводимо (Compose и Helm) и проходит Quality Gates.
- Архитектурная документация синхронизирована с реализацией (SSOT соблюдён).

## 8. Допущения и открытые вопросы

**Допущения** (подлежат подтверждению владельцем продукта):
- Аутентификация пользователей — через ведомственный Keycloak (OIDC).
- Телефонная платформа предоставляет запись разговора по URL для транскрибации.
- Целевая среда эксплуатации — Kubernetes в защищённом контуре.

**Открытые вопросы** (фиксируются как Backlog/Spike, решаются отдельными ADR/RFC):
- Перечень и контракты интеграций с внешними ведомственными системами.
- Требования по срокам хранения записей разговоров и ПДн.
- Требования по геораспределённому резервированию (DR-площадка).

Изменение Vision выполняется только через RFC с обновлением ADR Index и Architecture.md
(см. `docs/rfc/README.md`, `docs/governance/project-constitution.md`).
